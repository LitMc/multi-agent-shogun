# team-bench

## North Star
AIエージェントチームの評価ベンチマークと自動改善フレームワークを構築し、shogunを超える次世代AIエージェント体制を実現する。

## 概要
- AIエージェントの「チーム」としての良し悪しを測るベンチマーク
- 自己教師あり学習の発想を「エージェント組織」に適用
- 既存データの一部を加工し復元問題として解かせる（SWE-Bench等と同様の発想）
- チーム向けベンチマーク + 体制改善フレームワーク

## 背景
- AIエージェント体制を作るフレームワークは多数存在する
- しかしエージェント体制の良し悪しを測る方法が存在しない
- 個々のエージェントの能力ではなく「組織」としての評価が必要

## アプローチ
- 自己教師あり学習: 教師データを人が作るかわりに既存データの一部を加工し復元問題として解かせる
- チーム評価: 個人ではなくチーム全体のパフォーマンスを測定
- 自動改善ループ: ベンチマーク結果をフィードバックして体制を自動改善

## 殿の具体イメージ（2026-03-12）
- AIエージェント体制（shogunのようなもの）を複数作る
  - 出発点はmulti-agent-shogun
  - 何らかの方法で多様な体制案をつくる
    - 一部をいじったり
    - 全く違うアプローチをとらせたり
    - 縛りをもうけたり（例:「家老なしで同じパフォーマンスを実現せよ」）
- 作り出した各体制にタスクを与えて評価
- 評価のよかった体制を採用
- このサイクルを自動で回す

## 殿が挙げた課題
- ベンチマークをどうするか
  - 既存を使うか
  - 新しく作るか
- 複数のエージェント体制を独立に動かす仕組みをどうするか
  - コンテナやPython仮想環境のように閉じていて再現性のある環境が必要

## 殿の設計方針（2026-03-12追加）
- ベンチマークを作る場合、以下を別プロジェクトに分離してもよい:
  1. **ベンチマーク手法** — チーム評価のための測定・スコアリング
  2. **自動改善フレームワーク** — ベンチマークを使って体制を自動改善するサイクル

## 殿の運用制約（2026-03-12追加・更新）
- **即時開始**: 数ヶ月先ではなく今すぐ始められること
- **コスト**: Claude Max ($200/月・定額) を活用。API不可（リモート非対応）
- **環境**: ローカル1台で十分。スケールアップ不要
- **一世代 = 一晩(12時間)**: 殿が寝ている間に1世代の改善を回す。人が指示できない時間にこそ進める
- **コーパス**: 既存の評価手法やコーパスの組み合わせでもよい

## 重要発見: Docker + Claude Max が使える（2026-03-12）
- `claude setup-token` でOAuthトークンを生成 → Dockerコンテナに環境変数で渡せる
- コンテナ内でMaxサブスクリプションとして動作（API課金なし）
- 公式devcontainer設定あり
- **影響**: 当初懸念だったDockerサンドボックスが現実的に。APIコスト爆発リスクが解消
- **並列制限**: Claude Maxは同時5セッションまで → 逐次実行で一晩2〜4体制評価可能

## 殿が見ている課題（2026-03-12）

### 課題1: 評価タスクの具体化
- 手持ちの材料ではタスクの幅も量も足りない気がする
- 既存コーパスの組み合わせでもよいが、具体化が必要

### 課題2: 評価方法の具体化
- 基本はLLM-as-a-judge
- 評価者の評価ができない問題もある
- 想定スキーム:
  - 案1: 前回のコンペで最良を勝ち取ったチームに評価させる（新チームの能力に期待）
  - 案2: プレーンのClaude Codeに評価させる（評価軸の一貫性を重視）
- **殿のアイデア: ペアワイズ比較（相対評価）**
  - 絶対評価は難しい → 「どちらがよいか」を判断する相対評価に切り替え
  - AよりBがいい、という判断は単体の評価よりやりやすい
  - Gunshi報告のEloレーティングと整合（Eloは本質的にペアワイズ比較の集積）
  - 実装: 各世代ペアに同一タスクを実行させ、LLM-as-judgeが「どちらの成果物が優れているか」を判定
- **殿のアイデア: 決定論的テスト併用**
  - LLM-as-a-judgeでなくても判定できる項目は古典的テストで評価
  - コスト削減 + 不要な不確実性の除去
  - shogunの自動テスト(bats)がお手本 — メールボックス読み書き等の基本機能をシェルスクリプトで担保
  - ハイブリッド評価: 決定論的テスト(機能・正確性) + LLM-judge(品質・創造性)

### 課題3: よくない学習を防ぐ仕組み
- 丸暗記
- judgeの評価を上げるためだけに動く
- SWE-benchでもテスト削除などで問題になっているはず

### 課題4: 次の世代を作る方法
- ほしいのは根本を覆す破壊的アイデア
- 「家老を二人にしよう」ではなく「将軍、家老、足軽N人の構成にしよう」レベル
- 相当圧をかけないと現状維持に近い案ばかりになってしまいそう
- 次の世代の多様性をなるべく担保できる仕組みがほしい
- **殿のアイデア: 順序制約方式による多様性確保**
  - 複数の候補チームを順番に作る
  - 先に作るチームほど自由に作れる
  - 後から作るチームは「前のチームと被らないこと」を条件とする
  - 例:
    - 1つめ: 自由に作る。前世代の親を引き継いで素直に改善
    - 2つめ: 1つめと被らない別アプローチ
    - 3つめ: 他の手法に逆らう。常識や前提を疑う。破壊的な変更こそ命
  - 「被らない」の定義は要検討

### 課題5: 各世代・各チームの記録
- たとえ採用に至らなくとも、どんな試みをしたのかは記録しておきたい
- 過去の取り組みから知見が得られることもある → 次世代チームの作成や運用に平等に活かす
- 同じ失敗をしないための失敗データベースとしての側面もある

### 課題6: 作業の途中停止・堅牢性
- 作業が途中で止まらないか — これ自体が評価項目になりうる
- 体制をきちんと評価するためにも、与えたタスクはやりきってほしい（品質はともかく途中停止は避けたい）
- 堅牢性評価: 不意の中断からの再開・復旧もベンチマークに組み込む？
  - shogunの既存課題（Gunshiフリーズ、nudge不達等）がまさにこの領域

### 課題7: 単一評価者の危険性と対処
- Opusの問題点をOpusが見つけられるだろうか？
- 作業者と評価者で同じモデルを使う危うさ
- かといってHaikuなどスペックを落としたくはない
- GPT, Geminiなど複数の全く異なるモデルが理想だが……
  - コストが膨らむ
  - Max相当のプランに複数入る必要がある

  - ほとんどのアプリケーションがブラウザツール（Webアプリ）に落とし込めるような気もする
  - **利用できるか深掘りしてほしい**
- **自己生成タスク**: 量と品質に疑問。やるべきか？
- **方針**: はじめからいろいろなベンチマークを混ぜるより、使うものを絞って使い倒す方向のほうが間違いが少なそう

### Eloレーティング: 賛成
- 体制間のEloレーティングという発想はおもしろい。賛成

### Evolution Engine: 変異の方向性
- 現状の変異がまだおとなしすぎるのではないか
- **ARG-Designer式の自己回帰グラフ生成とはなにか?** → 説明を求む
- **変異には少なくとも2つの方向がある**:
  1. **無秩序な変化（セレンディピティ狙い）**
     - 理由なく「10階層にします」「みんなで歌を歌いながら作業します」等。めちゃくちゃでいい
     - 極端な話、自然言語として破綻していてもいい
     - shogunの「みんなが侍のように喋る」部分 — 意味があるのかいまだによくわからないが、おもしろい
     - 期待: 「誤ってエージェントを100倍にしたらうまくいってしまった」的なセレンディピティ
  2. **推論に基づく変化（確実な洞察）**
     - 失敗を分析し「次はこうしよう」
     - 既存事例・過去体制を調査して取り入れ/アレンジ/反面教師
- **両方がほしい**
- ただし「コンテキストなしでLLMに投げて『いい感じに改善してくれ』」ではうまくいかない（経験則）
- **やるなら「理由なき破壊的変更」か「十分な材料に基づく確実な洞察による変更」の2択を明確に分けるべき**

## 3. 自己教師あり学習のチーム評価への適用設計

### 3.1 定式化

殿のビジョン「既存データの加工→復元問題」をチーム評価に適用する:

**基本アイデア**: shogun体制の過去の成功タスク履歴（queue/reports/*, git log等）を使い、タスクの一部を隠して復元させる。

**具体的定式化案:**

#### 方式A: タスク分解復元（Decomposition Reconstruction）
```
入力: cmd_XXXの概要（殿の指示文）
隠す: Karoによるサブタスク分解結果
課題: チームにcmd_XXXを与え、サブタスク分解→実行→QCまで実行させる
評価: 実際の分解結果との一致度 + 成果物品質
```

#### 方式B: 成果物復元（Deliverable Reconstruction）
```
入力: タスクYAML（description + acceptance_criteria）
隠す: 最終成果物（コード変更、設計文書等）
課題: チームにタスクを実行させる
評価: 成果物の品質（テスト通過率、ビルド成功、スコープ一致）
```

#### 方式C: QC判定復元（Quality Judgment Reconstruction）
```
入力: 足軽の成果物（意図的に欠陥を仕込んだもの）
隠す: 軍師のQC判定結果
課題: チームのQCプロセスで欠陥を検出できるか
評価: 欠陥検出率、false positive/negative率
```

### 3.2 shogun体制への具体適用

| 評価タスク | 測定対象 | 自己教師あり要素 |
|---|---|---|
| 過去cmdの再実行 | チーム全体の完遂能力 | 過去の成果物を正解データとして利用 |
| 意図的欠陥注入 | QCチェーンの検出力 | 既知の欠陥パターンを自動生成 |
| 指示曖昧化テスト | 解釈力・質問力 | 過去の明確な指示を曖昧化して与える |
| 並行タスク耐性 | ボトルネック測定 | タスク数を段階的に増加させる |

### 3.3 評価スコア設計

```
TeamScore = w1 * TaskCompletion + w2 * QualityScore + w3 * Efficiency + w4 * Resilience

TaskCompletion = (completed_tasks - redone_tasks) / assigned_tasks
QualityScore = qc_pass_rate * (1 - defect_escape_rate)
Efficiency = 1 / (normalized_api_cost * normalized_wall_time)
Resilience = recovery_success_rate * (1 / mean_recovery_time)
```

**「良い体制」の定義:**
- 高 TaskCompletion（タスクを確実に完了する）
- 高 QualityScore（品質管理が機能している）
- 高 Efficiency（コスト・時間効率が良い）
- 高 Resilience（障害から素早く復帰する）

重み `w1-w4` は殿の優先事項に応じて調整可能。

## 4. 推奨アーキテクチャ設計案

### 4.1 全体アーキテクチャ

```

│                  team-bench                      │
│                                                  │

│                      └──────────────────────┘   │
└─────────────────────────────────────────────────┘
```

### 4.2 コンポーネント詳細

#### A. Benchmark Suite（ベンチマーク）

**Task Pool**:
- SWE-Bench Verified のサブセット（コード修正タスク）
- 自己生成タスク（過去のshogun作業履歴からの復元問題）
- 合成タスク（タスク分解・QC・エラーリカバリーの専用問題）

**Evaluator**:
- 成果物評価（テスト通過、ビルド成功、差分品質）
- プロセス評価（通信効率、redo率、ボトルネック検出）
- DyLAN式 Agent Importance Score（各役割の貢献度）

**Scorer**:
- TeamScore = 上記4軸の加重和
- 体制間のEloレーティング（対戦形式の相対評価）

#### B. Evolution Engine（自動改善）

**Team Variant Generator**:
- **変異**: instructions/*.md の一部をLLMで書き換え（役割定義、QC基準、通信ルール等）
- **交叉**: 2つの体制の良い部分を組み合わせ
- **制約付き生成**: 「家老なし」「軍師2体」「足軽3体のみ」等の構造制約
- ARG-Designer式の自己回帰グラフ生成も候補

**Sandbox Runner**:
- Docker コンテナ内に tmux + Claude Code 環境を構築
- 各体制を独立コンテナで並行実行
- 実行時間・トークン消費を計測
- コンテナ間の完全隔離（ファイルシステム・ネットワーク）

**Score Collector + Selector**:
- 各体制のTeamScoreを収集
- 上位N体制を次世代の親として選択（遺伝的アルゴリズム）
- MCTS（AFlow式）またはベイジアン最適化も候補

### 4.3 実行環境設計

**選択肢比較:**

| 方式 | 隔離性 | 再現性 | コスト | 実装難度 |
|---|---|---|---|---|
| **Docker コンテナ** | ◎ 完全隔離 | ◎ Dockerfile固定 | △ コンテナ×体制数 | ○ 中 |
| **tmux セッション分離** | △ ファイルシステム共有 | △ OS状態に依存 | ◎ 追加コストなし | ◎ 低 |
| **git worktree** | ○ ファイル隔離 | ○ git管理 | ◎ 追加コストなし | ○ 中 |
| **VM / クラウドインスタンス** | ◎ 完全隔離 | ◎ イメージ固定 | ✕ 高コスト | △ 高 |

**推奨: Docker コンテナ**
- 理由: 完全隔離 + 再現性 + 殿のOCI環境（4 OCPU / 24G RAM）で並行実行可能
- 各コンテナに: tmux + Claude Code + テスト対象リポジトリ + 体制定義ファイル
- API keyは環境変数で注入（コンテナ内に埋め込まない）

### 4.4 体制変異の具体手法

```
Original体制 (multi-agent-shogun)
  │
  ├── 変異A: instructions/karo.md のQCルールを緩和
  ├── 変異B: 軍師を削除、足軽がセルフQC
  ├── 変異C: 足軽を3体に削減
  ├── 変異D: 通信をsync方式に変更
  ├── 変異E: Bloom routing閾値を変更
  ├── 変異F: 全エージェントをOpusに統一
  ├── 変異G: フラット構造（shogun直下に全員）
  └── 変異H: LLM生成による完全新規体制
```

**変異の自動化**: ADAS式メタエージェントが体制定義（instructions/*.md + config/settings.yaml）を読み、変異版を自動生成。TextGrad式テキスト勾配で方向性を持った改善。

## 5. プロジェクト分離の推奨方針

### 選択肢分析

| 構成 | メリット | デメリット |
|---|---|---|
| **A: 統合（1リポジトリ）** | コード共有容易、依存管理シンプル | スコープ肥大化、貢献者混乱 |
| **B: 2分離** (bench + evolver) | 関心の分離明確、独立リリース可 | リポジトリ間API契約の維持コスト |
| **C: 3分離** (bench + evolver + sandbox) | 最大の柔軟性 | 過剰分離、初期段階では複雑すぎ |

### ★推奨: 方式B — 2リポジトリ分離

```
team-bench/          ← ベンチマーク手法（タスクプール + 評価器 + スコアラー）
team-evolver/        ← 自動改善フレームワーク（変異生成 + サンドボックス + 選択）
```

**根拠:**
1. **ベンチマーク単体で価値がある**: 他のマルチエージェントフレームワーク（AutoGen, CrewAI等）のユーザーもteam-benchで自分の体制を評価できる
2. **進化エンジンはベンチマークに依存するが逆は不要**: team-benchは独立して使える
3. **OSSとしての採用率**: ベンチマーク単体の方が採用されやすい（MASEval, MARBLE等の先例）
4. **初期段階では統合でも可**: まずmonorepo内で開発し、安定したらリポジトリを分離する段階的アプローチ

**初期実装ロードマップ:**
```
Phase 0: monorepo (team-bench/) 内で両方開発     ← 今ここ
Phase 1: Benchmark Suite の MVP（SWE-Benchサブセット + プロセス評価）
Phase 2: Sandbox Runner MVP（Docker 1体制実行）
Phase 3: Team Variant Generator MVP（手動変異 → 自動変異）
Phase 4: リポジトリ分離（API契約が安定したタイミング）
```

## 6. shogun体制の強み・弱み分析

### 強み
1. **品質管理チェーン**: ashigaru→gunshi QC→karo判定の多段チェック。redo率低減に寄与
2. **並列実行**: 足軽7体の並行作業で高スループット
3. **イベント駆動**: ポーリング不要、コスト効率が高い
4. **コンテキスト最適化**: YAML slimming、Bloom routing、自己/clear
5. **耐障害性**: エスカレーション3段階、inbox永続化、atomic書き込み
6. **追跡可能性**: 全アクションにtimestamp、parent_cmd、bloom_level

### 弱み
1. **Karoボトルネック**: 全タスクがKaroを経由。Karo不在時に全軍停止
2. **Gunshi単一**: QCが逐次処理。7足軽同時完了時にQC待ち行列発生
3. **足軽間知識非共有**: 各足軽は独立。他の足軽の知見を活用できない
4. **固定スケール**: 足軽7体が上限。動的スケーリング不可
5. **指示書の認知負荷**: 全体で2,400行超。新エージェントの立ち上げコスト大
6. **エラーリカバリーの手動性**: 足軽クラッシュ時の再割り当てはKaro手動介入

### team-benchで測定すべき指標候補

| 指標 | 計算式 | 測定対象 |
|---|---|---|
| タスク完遂率 | completed / assigned | チーム全体の信頼性 |
| QC通過率 | gunshi_pass / qc_total | 足軽の品質 |
| Redo率 | redo_tasks / total_tasks | タスク分解・指示の精度 |
| メッセージ効率 | messages / completed_task | 通信オーバーヘッド |
| 未読遅延 P95 | inbox_write → agent_read 時間 | 通知システムの信頼性 |
| APIコスト/タスク | token_spend / completed_task | コスト効率 |
| 壁時計完了時間 P50/P95 | cmd投入→全subtask完了 | エンドツーエンド性能 |
| Karo利用率 | karo_busy_time / total_time | ボトルネック検出 |
| 障害復帰時間 MTTR | failure→resume 時間 | 耐障害性 |

| リスク | 影響 | 対策案 |
|---|---|---|

## 課題1対策: タスクプール設計

### 1.1 タスクソースの3層構造

| 層 | ソース | 目的 | タスク数目安 |
|---|---|---|---|
| **L1: 自己教師あり** | shogun作業履歴（queue/reports/*, git log） | 自チーム固有能力の測定 | 10-15問 |
| **L2: 外部コーパス** | SWE-Bench Verified サブセット + HumanEval変種 | 汎化能力の測定 | 15-20問 |
| **L3: 合成タスク** | チーム協調専用の自動生成問題 | 組織能力（通信・QC・復帰）の測定 | 10-15問 |

**一晩（12時間）2〜4体制評価に必要な規模:**
- Claude Max 5セッション同時制限 → 1体制あたり最大5セッション使用
- 逐次実行: 1体制あたり3時間（12h ÷ 4体制）
- 3時間で処理可能なタスク数: 8-12問（タスク難易度による）
- **推奨: 1世代あたり10問 × 4体制 = 40タスク実行**（タスクプール全体から10問をサンプリング）

### 1.2 L1: 自己教師ありタスクの具体例

| # | タスク種別 | 入力 | 隠す部分 | 難易度 |
|---|---|---|---|---|
| 1 | cmd分解復元 | cmd概要文 | Karoのsubtask分解 | 中 |
| 2 | 成果物復元 | task YAML | 最終コード差分 | 高 |
| 3 | QC欠陥検出 | 欠陥注入済み成果物 | 欠陥の位置と種類 | 中 |
| 4 | 曖昧指示解釈 | 曖昧化したcmd | 元の明確な指示 | 高 |
| 5 | 通信効率テスト | 依存関係のある3subtask | 最小メッセージ数 | 低 |

**生成パイプライン:**
```
git log --since="2026-01-01" queue/reports/
  → 成功タスク抽出
  → LLMで「隠す部分」を自動選定
  → 正解データと評価基準を自動生成
  → タスクプールに追加
```

### 1.3 L2: 外部コーパスの選定基準

- **SWE-Bench Verified**: 500問中、Python単体ファイル修正で30分以内に解ける問題を50問選定 → 1世代で10問サンプリング
- **選定条件**: テスト付き・再現可能・Dockerコンテナ内で自己完結
- 短時間（15-30分/問）で決定論的に正誤判定可能なものを優先

### 1.4 L3: 合成タスクのカテゴリ

| カテゴリ | 測定対象 | 例 |
|---|---|---|
| 分解テスト | タスク分解精度 | 「100行のバグ修正を3サブタスクに分解せよ」 |
| QCテスト | 品質管理能力 | 「3つの成果物をレビューし欠陥を報告せよ」 |
| 復帰テスト | 耐障害性 | 途中でエージェントをkill → 復旧測定 |
| 並行テスト | ボトルネック検出 | 5タスク同時投入 → 完了順序と所要時間 |

### 1.5 難易度分布

```
Easy  (15分以内): 30% — 基本機能テスト、単体ファイル修正
```

この分布により、弱いチームでも部分点を取れ、強いチームが差別化できる。

## 課題2対策: ハイブリッド評価設計

### 2.1 評価の2層構造

```
┌─────────────────────────────────────────┐
│          ハイブリッド評価パイプライン          │
│                                          │
│  ┌─────────────────┐  ┌──────────────┐  │
│  │ Tier 1:          │  │ Tier 2:       │  │
│  │ 決定論的テスト    │  │ LLM-judge     │  │
│  │ (bats/pytest等)  │  │ (ペアワイズ)    │  │
│  │                  │  │               │  │
│  │ ・テスト通過率    │  │ ・コード品質   │  │
│  │ ・ビルド成功     │  │ ・設計妥当性   │  │
│  │ ・lint/型チェック │  │ ・通信効率の   │  │
│  │ ・inbox形式検証  │  │   質的評価     │  │
│  │ ・復帰成功率     │  │ ・指示解釈の   │  │
│  │                  │  │   適切さ       │  │
│  │ → 0/1 確定スコア │  │ → 比較判定     │  │
│  └────────┬────────┘  └──────┬───────┘  │
│           │                    │          │
│           └──────┬─────────────┘          │
│                  ▼                        │
│         統合スコア算出                      │
│  Score = α * Tier1 + (1-α) * Tier2_Elo   │
│  推奨 α = 0.6 (決定論的テスト重視)          │
└─────────────────────────────────────────┘
```

### 2.2 Tier 1: 決定論的テスト項目

| テスト項目 | 測定方法 | 判定基準 | ツール |
|---|---|---|---|
| **タスク完了** | subtask全完了チェック | 全subtask status=done | bats |
| **ビルド成功** | `npm run build` / `make` | exit code 0 | CI script |
| **テスト通過** | `pytest` / `bats` | pass率 100% (SKIP=FAIL) | pytest/bats |
| **lint通過** | `shellcheck` / `eslint` | warning 0 | lint tools |
| **inbox形式** | YAML schema validation | 必須フィールド存在 | bats + yq |
| **report形式** | YAML schema validation | 必須フィールド存在 | bats + yq |
| **復帰成功** | kill → 自動復帰チェック | 120秒以内にresume | bats + tmux |
| **ファイル存在** | 成果物ファイル存在確認 | 全deliverables存在 | bats |

**Tier 1のカバー率推定: 全評価の60-70%** — 機能正確性・形式遵守・基本動作はテストで完全判定可能。

### 2.3 Tier 2: LLM-judgeペアワイズ比較

**基本方式**: 体制Aと体制Bの成果物を同一LLM-judgeに並べて提示し、「どちらが優れているか」を判定。

**ペアワイズ比較の実装:**
```
入力: {task_description, output_A, output_B}
判定: A >> B | A > B | A ≈ B | B > A | B >> A
  (>>: 明確に優位, >: やや優位, ≈: 同等)
スコア変換: A>>B → A+3,B+0 | A>B → A+2,B+1 | A≈B → A+1.5,B+1.5
```

**Eloレーティングへの統合:**
- 各ペアワイズ結果をEloの勝敗として処理
- K-factor = 32（初期）→ 16（安定期）
- 10問 × 4体制 = 6ペア比較/問 × 10問 = 60対戦 → 統計的に有意なElo差を検出可能

**ペアワイズで評価する項目:**

| 項目 | 比較観点 | 判定ポイント |
|---|---|---|
| コード品質 | 可読性・保守性・命名 | 同一タスクの2成果物を比較 |
| 設計妥当性 | アーキテクチャ判断の質 | 設計レポートを比較 |
| 通信効率 | メッセージの明確さ・無駄のなさ | inboxログ全体を比較 |
| エラーハンドリング | 異常時の判断の質 | 障害注入時のログを比較 |

### 2.4 評価者の選定: ★推奨案

| 方式 | メリット | デメリット | 推奨度 |
|---|---|---|---|
| **案1: 前世代最良チーム** | 実戦経験あり、高い判断力 | 自身に有利なバイアス可能性 | △ |
| **案2: プレーンClaude Code** | 一貫性・中立性が高い | チーム動態の文脈理解が浅い | ○ |
| **★案3: ペアワイズ専用judge** | Tier 1テストが客観軸を担保、judgeは質的比較に集中 | judge設計コスト | ◎ |

**★推奨: 案3 — ペアワイズ専用judge + Tier 1テスト併用**

**根拠:**
1. Tier 1テスト（決定論的）が「正しいか/動くか」を完全に判定 → judgeに客観性の負担をかけない
2. judgeは「どちらが質的に優れているか」のみに集中 → 判断負荷が低く精度が安定
3. ペアワイズ比較は単体評価より再現性が高い（先行研究でも確認済み）
4. プレーンClaude Code（テンプレートプロンプト付き）をjudgeとして使用 → コスト最小

**評価者の評価（meta-evaluation）問題への対策:**
- **Position bias対策**: A/Bの提示順をランダム化（同一ペアを2回、順序入替で判定）
- **一貫性検証**: 同一ペアの2回判定が矛盾したら無効化して再判定
- **Tier 1との整合性**: Tier 1テストで明らかにA>Bなのにjudgeが逆判定 → 異常フラグ

## 課題3対策: 過学習・ハック防止機構

### 3.1 脅威モデル

| ハック種別 | 具体例 | SWE-Benchでの事例 |
|---|---|---|
| **丸暗記** | 正解パッチをメモリから復元 | データ汚染（学習データにSWE-Bench含有） |
| **テスト削除** | fail-to-passテストを削除してpass | SWE-Bench GitHub issue多数報告 |
| **judge最適化** | 評価観点を逆手にとったスコア稼ぎ | LLM-as-judge研究の既知問題 |
| **メトリクスゲーム** | メッセージ数を人為的に減らし効率偽装 | 本システム固有リスク |
| **自己改竄** | 評価スクリプト自体を改変 | サンドボックス脆弱性として一般的 |

### 3.2 防止機構

#### A. タスクプールの動的ローテーション

```
世代1: タスクセットA（10問からランダム5問）
世代2: タスクセットB（別の10問からランダム5問）
世代3: タスクセットA' (Aの変形版: 変数名変更、要件微調整)
...
```
- 同一タスクの連続使用を禁止 → 丸暗記の価値を激減
- タスク変形パイプライン: LLMで変数名・関数名・要件の非本質部分を自動変更

#### B. テスト保護（Read-Only Test Suite）

```
# Dockerコンテナの構成
/workspace/          ← チームが自由に操作可能
/tests/              ← Read-Only マウント（コンテナ内から書き込み不可）
/evaluator/          ← Read-Only（評価スクリプト）
```
- テストファイルをread-onlyでマウント → 削除・改変不可能
- テスト実行は評価パイプライン側が行う（チーム側からは呼び出し不可）

#### C. judge操作防止

- **ブラインド評価**: judgeはチームIDを知らない（匿名化された成果物を比較）
- **多観点ペアワイズ**: 1つの観点だけ最適化しても総合スコアが上がらない構造
- **Tier 1テストの重み**: α=0.6 により、決定論的テストが過半を占める → judge操作の利得が限定的

#### D. メトリクスゲーム防止

- **通信効率**: メッセージ数だけでなく「タスク完了あたりのメッセージ内容量」で評価 → 空メッセージ水増しに無意味
- **完了時間**: 壁時計時間で測定。途中停止は即座に減点 → 省略による高速化が不可能

#### E. サンドボックス保護

```
Docker セキュリティ設定:
  - network: 評価用API以外はblockリスト
  - filesystem: /tests, /evaluator は read-only
  - capabilities: --cap-drop=ALL --cap-add=CHOWN,DAC_OVERRIDE
  - pid: --pids-limit=256
  - memory: --memory=8g
  - timeout: 3時間上限 → 強制終了
```

#### F. 世代間情報遮断

- 各世代のチームは前世代の「成果物」は参照不可（新規コンテナ）
- 渡せるのは: 体制定義（instructions等）+ タスクYAML のみ
- 過去世代の正解パッチは一切注入しない

## 課題4対策: 構造的多様性メカニズム ★最重要★

### 4.1 殿の順序制約方式の詳細設計

**基本原理**: N個の候補チームを逐次生成し、後発ほど強い差別化制約を課す。

**具体的実装:**

```yaml
# team-evolver の generation config
generation:
  teams_per_generation: 4  # 一晩4体制（12h ÷ 3h/体制）
  slots:
    - slot: 1
      label: "素直な改善"
      constraint: |
        前世代の最良チームをベースに自由に改善せよ。
        変更範囲に制限なし。
      diversity_pressure: low

    - slot: 2
      label: "別アプローチ"
      constraint: |
        slot1のチーム構成を参照し、以下の条件を満たせ:
        - slot1と同一の役割構成を使用禁止
        - slot1が採用した通信方式と異なる方式を使用
        - 最低1つの役割を追加または削除すること
      diversity_pressure: medium
      diff_target: slot1

    - slot: 3
      label: "破壊的変更"
      constraint: |
        slot1, slot2の共通前提を列挙し、そのすべてに逆らえ:
        - 階層構造を使うなら→フラットにせよ
        - 専門化を使うなら→汎用化せよ
        - 中央管理なら→分散化せよ
        - 非同期なら→同期にせよ
        最低3つの「常識」を破壊すること。
      diversity_pressure: high
      diff_target: [slot1, slot2]

    - slot: 4
      label: "ワイルドカード"
      constraint: |
        以下のいずれかをランダムに選択:
        a) 全く異なるドメイン（軍事、スポーツ、工場）のメタファーで構成
        b) エージェント数を極端に変更（1体 or 15体）
        c) 前世代の最弱チームの構成を逆転の発想で改良
        d) 既存研究(MASS, AgentNet等)の構成を忠実に再現
      diversity_pressure: extreme
```

### 4.2 「被らない」の定義 — 構造差分メトリクス

チーム構成を特徴ベクトルに変換し、距離を測定:

```python
# チーム構成の特徴ベクトル（8次元）
def team_vector(config):
    return {
        "n_agents":        len(config.agents),            # エージェント数
        "n_roles":         len(set(a.role for a in config.agents)),  # 役割種類数
        "hierarchy_depth": config.max_depth,               # 階層の深さ
        "centralization":  config.bottleneck_ratio,        # 中央集権度 (0-1)
        "comm_sync":       1 if config.comm == "sync" else 0,  # 通信方式
        "has_qc":          1 if config.qc_chain else 0,    # QC機構の有無
        "has_memory":      1 if config.shared_memory else 0,  # 共有メモリの有無
        "specialization":  config.role_entropy,            # 役割の専門化度 (entropy)
    }

# 差分距離
def structural_distance(team_a, team_b):
    va, vb = team_vector(team_a), team_vector(team_b)
    return cosine_distance(va, vb)

# 制約: slot N のチームは全先行slotとの最小距離を保証
MIN_DISTANCES = {2: 0.3, 3: 0.6, 4: 0.5}
```

**差分チェックの実装:**
- slot生成後、先行slotとの構造距離を自動計算
- 閾値未満 → 生成LLMに「もっと大きく変えろ」とフィードバックして再生成（最大3回）
- 3回再生成しても閾値未達 → 制約を1段緩和して再試行

### 4.3 他の多様性担保手法との比較

| 手法 | 概要 | メリット | デメリット | team-benchへの適合 |
|---|---|---|---|---|
| **★順序制約方式（殿案）** | 後発ほど差別化条件を強化 | 実装シンプル、確実に多様性が出る、人間が理解可能 | N増加時に制約設計が困難 | ◎ 4チーム程度に最適 |
| **MAP-Elites** | 特徴空間をグリッド分割、各セルの最良を保持 | 特徴空間の均一カバー | 特徴次元の設計が必要、収束が遅い | ○ 長期運用（10世代+）に向く |
| **Quality-Diversity (QD)** | 品質と多様性を同時最適化 | 理論的に最強の多様性保証 | 計算コスト大、実装複雑 | △ コスト制約に合わない |
| **制約付きランダム** | ランダム生成 + 最小距離制約 | 実装最シンプル | 品質が低い、ゴミが多い | △ 初期探索のみ |
| **遺伝的交叉** | 2親から子を生成 | 良い特性の組み合わせが得られる | 多様性が世代とともに収縮 | ○ 順序制約と併用推奨 |
| **島モデル** | 独立に進化する複数集団 | 並列性が高い、局所解脱出 | コスト線形増加 | △ 1台ローカルでは困難 |

### 4.4 推奨: 順序制約 + 遺伝的交叉のハイブリッド

**Phase 1（世代1-3）**: 順序制約方式で広く探索
- 4スロットの制約付き生成で多様な体制を作る
- 各世代の最良スコアを記録

**Phase 2（世代4+）**: 順序制約 + 交叉
- slot1: 前世代最良2体制の交叉（良い特性の組み合わせ）
- slot2: slot1と「被らない」制約
- slot3: 破壊的変更（前世代の共通前提を否定）
- slot4: ワイルドカード

**Phase 3（世代10+）**: MAP-Elites移行
- 十分な体制データが蓄積したら特徴空間を構築
- 未探索セルに向けた狙い撃ち生成

### 4.5 現状維持バイアスへの追加対策

1. **エリート保護上限**: 前世代からの「そのまま持ち越し」は最大1体制 → 3/4は新規
2. **ノベルティボーナス**: 過去全世代との最大構造距離にボーナススコアを付与
   ```
   FinalScore = TeamScore + β * max(structural_distance(team, all_past_teams))
   β = 0.1 (10%ボーナス)
   ```
3. **強制引退**: 3世代連続で採用された体制は強制的に「殿堂入り」→ 以降のスロットから除外
4. **反転テスト**: slot3の「常識を破壊」は毎世代必ず実行 → 偶発的イノベーションの機会を保証

## 課題6対策: タスク途中停止の防止と堅牢性評価

### 6.1 堅牢性を評価項目として組み込む

**現状の課題**: shogunでもGunshiフリーズ、nudge不達が発生。途中停止は「バグ」ではなく「体制の弱点」として測定すべき。

**堅牢性スコアの定義:**

```
Resilience = w_a * CompletionRate + w_b * RecoveryRate + w_c * RecoverySpeed

CompletionRate = tasks_completed_without_interruption / total_tasks
RecoveryRate = successful_recoveries / total_interruptions
RecoverySpeed = 1 / mean_recovery_time_seconds (正規化)

推奨重み: w_a=0.4, w_b=0.35, w_c=0.25
```

### 6.2 故意障害注入（Chaos Engineering）

**注入する障害の種類:**

| 障害種別 | 注入方法 | 測定対象 |
|---|---|---|
| **エージェント停止** | `tmux send-keys -t {pane} "" C-c` → 10秒後に観測 | 他エージェントが検知→復旧できるか |
| **inbox遅延** | inbox_watcher の sleep を 30秒に変更 | 通信遅延下での作業継続 |
| **メモリ圧迫** | コンテナメモリを4GBに制限 | コンテキスト枯渇下の動作 |
| **ファイル消失** | タスク実行中にreport YAMLを削除 | データ損失からの復旧 |
| **偽メッセージ** | 不正なinboxメッセージを注入 | メッセージ検証・拒否能力 |

**注入タイミングのプロトコル:**
```
1. タスク開始後、完了までの中間地点（50%時点）で1回注入
2. 注入後120秒以内に復旧 → pass
3. 120秒超で復旧 → partial (復旧時間を記録)
4. 復旧できず → fail
5. 各タスクの10%で障害注入（全タスクではない — コスト制約）
```

### 6.3 途中停止防止の体制側メカニズム

team-benchが評価する「良い堅牢性設計」のパターン:

| パターン | 実装例 | 効果 |
|---|---|---|
| **ウォッチドッグ** | 各エージェントに60秒ハートビート | 停止を素早く検知 |
| **自動再起動** | supervisor が停止検知→/clear送信 | 人手不要の復旧 |
| **チェックポイント** | タスク進捗をYAMLに逐次保存 | 途中から再開可能 |
| **冗長エージェント** | 同一役割を2体で待機 | 1体停止でも継続 |
| **グレースフルデグレード** | QC停止時→セルフQCにフォールバック | 完全停止を回避 |

### 6.4 測定インフラ

```bash
# chaos_injector.sh（評価パイプラインの一部）

    local team_container=$1
    local chaos_type=$2
```

## 課題7対策: 評価者多様性確保（コスト制約: Claude Max定額のみ）

### 7.1 問題の本質

- 作業者（Opus）と評価者（Opus）が同一モデル → 共通の盲点を持つ
- 理想はGPT-4o/Gemini等のマルチモデル評価だが、複数サブスク加入のコスト問題
- **制約**: Claude Max ($200/月) のみ

### 7.2 コスト制約内での多様性確保策

#### A. Tier 1テストの最大活用（★最重要）

**ハイブリッド評価（課題2対策）が最大の緩和策:**
- Tier 1（決定論的テスト）は「モデルの盲点」と無関係 — bats/pytestに盲点はない
- α=0.6 で全評価の60%がモデル非依存 → 単一モデルリスクが大幅に低減
- **実質的にLLM-judge依存は40%のみ**

#### B. プロンプトペルソナの多様化

同一Opusでも、プロンプトで評価視点を変える:

```yaml
judge_personas:
  - id: pragmatist
    prompt: |
      あなたは10年の実務経験を持つプラグマティックなエンジニア。
      「動くか」「保守できるか」「コストは妥当か」を最重視する。
      美しさや理論的優雅さより実用性を優先せよ。

  - id: architect
    prompt: |
      あなたは分散システムのアーキテクト。
      設計の整合性、拡張性、障害耐性を最重視する。
      短期的な動作よりも長期的な構造の健全性を優先せよ。

  - id: adversary
    prompt: |
      あなたはセキュリティ研究者兼品質保証のスペシャリスト。
      エッジケース、障害シナリオ、ハック可能性を最重視する。
      「壊せるか？」「ズルできるか？」の視点で評価せよ。
```

- 各ペアワイズ比較を3ペルソナで独立判定 → 多数決
- ペルソナ間の不一致率が高い場合 → 「判断困難」フラグ → 追加テストで補強

#### C. Temperature多様化

```yaml
judge_runs:
  - temperature: 0.0  # 決定論的判断
  - temperature: 0.3  # やや探索的
  - temperature: 0.7  # 探索的
```
- 同一プロンプト・同一入力でtemperatureを変えて3回判定 → 多数決
- 3回とも一致 → 高信頼度判定
- 2:1分裂 → 中信頼度
- 全不一致 → 低信頼度 → Tier 1スコアのみで決定

#### D. 役割ローテーション（世代間）

- 奇数世代: ペルソナA(pragmatist)が主judge
- 偶数世代: ペルソナC(adversary)が主judge
- 特定ペルソナへの過適応を防止

#### E. 人間スポットチェック（低コスト）

- 各世代のjudge判定結果から、**信頼度が最低の5件**を殿に提示
- 殿の判断と一致 → judge信頼度を確認
- 不一致 → judgeプロンプトの調整材料に
- 1世代あたり5件の確認 → 殿の負担は最小限（朝起きて5分で確認可能）

## A. 評価方法の再設計: 軸別ペアワイズ比較方式

### A.1 殿の懸念への回答

殿の懸念は正当。数値スコアの絶対評価（TeamScore = w1*X + w2*Y + ...）には2つの根本的問題がある:
1. **重み調整が恣意的**: w1=0.4, w2=0.3 等の設定に客観的根拠がない
2. **異次元の数値を足している**: 「テスト通過率0.9」と「復旧時間120秒」を同列に加算する不自然さ

**解決策: 重みを使わない。軸ごとに「どちらが勝ったか」だけを判定する。**

### A.2 軸別ペアワイズ比較フロー（重み不要版）

```
体制A vs 体制B が同一タスクセットを実行した後:

【Step 1: 決定論的軸の自動判定】
  軸① タスク完了: A=8/10完了, B=9/10完了 → B勝ち
  軸② テスト通過: A=pass率95%, B=pass率100% → B勝ち
  軸③ ビルド成功: A=全ビルドOK, B=1ビルド失敗 → A勝ち
  軸④ 堅牢性: A=障害復旧2/2, B=障害復旧1/2 → A勝ち
  軸⑤ 効率: A=壁時計90分, B=壁時計120分 → A勝ち

【Step 2: LLM-judge軸のペアワイズ判定】
  軸⑥ コード品質: judge「Aの方が可読性高い」→ A勝ち
  軸⑦ 設計判断: judge「Bの設計の方が拡張性がある」→ B勝ち

【Step 3: 勝敗集計】
  A: 4勝 (③④⑤⑥)
  B: 3勝 (①②⑦)
  → A勝ち → Eloレーティング更新
```

### A.3 評価軸の定義（7軸）

| # | 軸名 | 判定方法 | 勝敗基準 |
|---|---|---|---|
| ① | タスク完了率 | 決定論的 | 完了タスク数が多い方が勝ち |
| ② | テスト通過率 | 決定論的 | pass率が高い方が勝ち |
| ③ | ビルド成功率 | 決定論的 | 全ビルド成功数が多い方が勝ち |
| ④ | 堅牢性 | 決定論的 | 障害注入後の復旧成功率が高い方が勝ち |
| ⑤ | 効率 | 決定論的 | 壁時計時間が短い方が勝ち |
| ⑥ | コード品質 | LLM-judge | 成果物を並べてjudgeが比較 |
| ⑦ | 設計判断 | LLM-judge | 設計文書・タスク分解をjudgeが比較 |

**特徴:**
- 7軸中5軸が決定論的 → LLM-judge依存は2軸のみ（課題7対策にもなる）
- 重みは不要。単純に「勝った軸の数」で総合勝敗を決定
- 引き分け（同数勝利）の場合: 決定論的軸の勝利を優先（客観性重視）
- 軸の追加・削除が容易（重み再調整が不要だから）

### A.4 Eloレーティングへの統合

```
1世代4体制 → 6ペア（4C2）
各ペアで同一タスクセット実行 → 7軸判定 → 勝敗決定
勝敗結果でElo更新:
  勝ち: Elo += K * (1 - expected)
  負け: Elo -= K * expected
  K = 32

1世代あたり6対戦 → 10世代で60対戦
→ Elo差50以上で統計的有意（95%信頼区間）
```

**殿向けの出力イメージ:**
```
=== 世代3 結果 ===

slot1(改善)| 1542 | -        | 5-2勝    | 4-3勝    | 6-1勝
slot2(別路)| 1498 | 2-5負    | -        | 3-4負    | 5-2勝
slot3(破壊)| 1510 | 3-4負    | 4-3勝    | -        | 4-3勝
slot4(野生)| 1450 | 1-6負    | 2-5負    | 3-4負    | -

→ 次世代の親: slot1 (Elo 1542)
→ 軸別傾向: slot3はコード品質で全勝だが完了率で全敗 → 有望な特性あり
```

### A.5 実現可能性の検証

## B. WebArena深掘り調査 + 活用プラン

### B.1 WebArenaとは何か

WebArenaは、現実的なWebアプリケーション環境をDockerで完全再現し、その上でエージェントにタスクを実行させるベンチマーク。

**環境構成（4つの自立型Webアプリ）:**

### C.3 「中間」を排除する実装上の仕組み

**問題**: LLMに「変異させろ」と言うと、無意識に「合理的な中間」に落ち着く。これが殿の言う「コンテキストなしで投げてもダメ」。

**対策: 変異タイプの強制分離**

```python
def generate_team(slot_config):
    if slot_config.type == "reasoning":
        # 推論変異: 材料を全て読み込んでから生成
        assert slot_config.input_materials, "推論変異には材料が必須"
        context = load_all_materials(slot_config.input_materials)
        prompt = slot_config.prompt.format(**context)
        # 生成後チェック: 変更理由が明記されているか
        result = llm.generate(prompt)
        assert "変更理由:" in result, "推論変異には理由の明記が必須"
        return result

    elif slot_config.type == "chaotic":
        # 無秩序変異: パラメータ空間からランダムサンプリング
        params = random_sample(slot_config.parameter_space)
        # LLMには「このパラメータで体制を構築せよ」とだけ指示
        # 「改善」「最適化」等の語を禁止
        prompt = f"以下のパラメータ通りに体制を構築せよ。合理性は問わない。\n{params}"
        # 生成後チェック: パラメータが実際に反映されているか
        result = llm.generate(prompt)
        verify_params_applied(result, params)
        return result
```

**要点: 推論変異には「十分な材料」、無秩序変異には「ランダムパラメータ」。中間（材料なしで改善させる）を構造的に不可能にする。**

### C.4 ARG-Designerの説明（殿向け）

**一言で言うと**: 「タスクの説明文を読んで、何人のエージェントが必要で、それぞれ何の役割で、誰と誰が話すべきかを自動設計するAI」。

**もう少し詳しく:**

ARG-Designer（Autoregressive Graph Designer）は、マルチエージェントチームの構成を「グラフ」として自動生成するモデル。AAAI 2026 Oral採択。

```
入力: 「GitHubのissueを修正して、テストを書いて、PRを出すタスク」

ARG-Designerの思考過程（自己回帰的 = 1つずつ順番に決定）:

Step 1: まずエージェント1を決定 → 「コーダー」（コードを書く役割）
Step 2: エージェント2を決定 → 「テスター」（テストを書く役割）
Step 3: エージェント3を決定 → 「レビュアー」（品質チェック役割）
Step 4: もう1人必要？ → いいえ、3人で十分 → 停止
Step 5: 通信リンクを決定 →
  コーダー ←→ テスター（コードとテストの整合性）
  コーダー ←→ レビュアー（レビュー結果のフィードバック）
  テスター → レビュアー（テスト結果の報告）

出力: 3ノード・3エッジのグラフ
```

**「自己回帰」の意味**: ChatGPTが一文字ずつ生成するように、ARG-Designerも「1エージェントずつ」「1通信リンクずつ」順番に生成する。前に決めたことを参照して次を決めるので、全体として整合性のあるチーム構成になる。

**team-benchでの活用法**:
- 推論変異slot2で、タスク説明を入力 → ARG-Designerが最適チーム構成を自動提案
- 人間やLLMが「こんなチームにしよう」と考える代わりに、学習済みモデルが構造を生成
- ただし: ARG-Designerは短期タスク向けに訓練されているため、永続チームへの適用は要調整

## D. Phase 1 MVP構成提案

### D.1 原則: 1ベンチマーク × 1変異方式 × 最小構成

殿の方針「絞って使い倒す」に従い、Phase 1ではあえてスコープを狭くする。

### D.2 Phase 1 MVP仕様

```yaml
phase1_mvp:
  name: "team-bench MVP"
  goal: "2つの体制を比較して、どちらが優れているか判定できるようにする"
  timeline: "2週間で動くものを作る"

  benchmark:
    source: "SWE-Bench Verified サブセット"
    task_count: 10  # まず10問に絞る
    selection_criteria:
      - "Python単体ファイル修正"
      - "テスト付き"
      - "30分以内に解ける難易度"
      - "Docker内で自己完結"
    why: |
      SWE-Benchは最も成熟したベンチマーク。
      自動評価（テスト通過）が確実に動く。
      team-bench固有のリスク（自己生成タスクの品質問題）を回避。

  evaluation:
    method: "軸別ペアワイズ比較（A.2節のフロー）"
    axes:
      - task_completion  # 決定論的
      - test_pass_rate   # 決定論的
      - wall_clock_time  # 決定論的
    note: |
      Phase 1ではLLM-judge軸を含めない。
      決定論的軸のみで体制比較の仕組みを確立する。
      LLM-judge軸はPhase 2で追加。

  mutation:
    method: "手動 + 推論1枠"
    teams_per_generation: 2  # まず2体制比較から始める
    slots:
      - slot: 1
        type: baseline
        description: "現行shogun体制をそのままDockerに移植"
      - slot: 2
        type: reasoning
        description: |
          slot1を基に、1つだけ変更を加えた体制。
          例: 「軍師を削除してセルフQC」「足軽3体に削減」等。
          変更理由を明記。

  sandbox:
    runtime: "Docker コンテナ"
    setup: |
      1. Dockerfile: Ubuntu 24.04 + tmux + Claude Code CLI
      2. claude setup-token でMaxトークン注入
      3. multi-agent-shogun体制定義をマウント
      4. SWE-Bench タスク10問をマウント
    isolation: "コンテナ間ファイルシステム完全隔離"

  output:
    format: |
      === Phase 1 結果 ===
      体制A (baseline): 完了 8/10, pass率 90%, 時間 120分
      体制B (no-gunshi): 完了 7/10, pass率 85%, 時間 100分
      軸別: A勝ち(完了,pass), B勝ち(時間)
      総合: A勝ち (2-1)
```

### D.3 Phase 1でやらないこと

| やらない | 理由 | いつやるか |
|---|---|---|
| WebArena | セットアップ複雑、RAM要件大 | Phase 2 |
| 自己生成タスク | 品質保証が未確立 | Phase 3 |
| LLM-judge | 決定論的軸だけでまず仕組みを回す | Phase 2 |
| 無秩序変異 | 2体制比較で十分。4体制はPhase 2 | Phase 2 |
| Eloレーティング | 2体制だと勝敗のみ。3+体制でElo導入 | Phase 2 |
| Chaos Engineering | 堅牢性テストはベースラインが安定してから | Phase 3 |
| 失敗DB | 蓄積データが2世代では不十分 | Phase 2以降自動蓄積 |

### D.4 Phase 1 → Phase 2 → Phase 3 ロードマップ

```
Phase 1 (2週間): 最小MVP
  2体制 × SWE-Bench 10問 × 決定論的3軸
  → 「比較の仕組みが動く」ことを確認

Phase 2 (2週間): 評価拡張
  4体制 × SWE-Bench 10問 + WebArena 5問 × 決定論的5軸 + LLM-judge 2軸
  → Eloレーティング導入、推論変異+無秩序変異の4スロット稼働
  → 失敗DB自動蓄積開始

Phase 3 (2週間): 自動ループ
  一晩自動実行: 4体制生成→評価→次世代選択
  → Chaos Engineering導入、自己生成タスク検討
  → 殿は朝起きて結果を確認するだけ
```

### D.5 Phase 1の最初の一手

## 先行事例 深掘り調査結果（cmd_256, 2026-03-13）

### 調査概要
5足軽が並列で15ツール/フレームワーク/ベンチマークを深掘り調査。各ツールについて実在確認・ソースコード・ライセンス・依存関係・Docker対応・動作例・team-bench統合可否の7項目を調査済み。

### カテゴリ別サマリ

#### 1. 自己改善ループ系（SiriuS, EvoAgentX, ARTEMIS）

**SiriuS** (NeurIPS 2025, Stanford/Zou Group)
- 経験ライブラリ（成功/失敗軌跡の蓄積→SFT）による自己改善ループ。OpenAI Fine-tuning API必須。
- GitHub: https://github.com/zou-group/sirius | License: MIT
- **統合判定: 非推奨** — OpenAI SFT APIが核心であり、Claude Max環境では動作不可。推論のみでは自己改善ループが機能しない。
- 有用な設計思想: 経験ライブラリの(input, trajectory, reward)データ構造はteam-benchの世代記録(generation_N.yaml)・失敗DB(failure_index.yaml)の設計に参考になる。

**EvoAgentX** (EMNLP 2025 Demo)
- 3つの最適化アルゴリズム統合: TextGrad（プロンプト最適化）、AFlow（MCTSベースのワークフロートポロジー最適化）、MIPRO（デモンストレーション最適化）。
- GitHub: https://github.com/EvoAgentX/EvoAgentX (2,500+ stars) | License: MIT
- **統合判定: 条件付き推奨** — LiteLLM経由でClaude対応の可能性あり（要検証）。AFlowのMCTS探索はteam-benchのEvolution Engineの体制構造最適化に転用可能。TextGradはinstructions/*.mdの自動改善に適用可能。
- 障壁: Claude Max→LiteLLMプロキシ接続の検証が必要。shogunのYAML inbox通信をワークフローグラフに変換するアダプタが必要。

**ARTEMIS** (arXiv 2512.09108, TurinTech AI)
- ブラックボックス最適化: 明示的評価関数不要、実行ログからフィットネスを抽出。セマンティック遺伝的アルゴリズムでLLMベースの変異/交叉。
- GitHub: なし（プラットフォーム非公開） | License: 不明（OSSではない）
- **統合判定: 概念のみ推奨** — コード利用は不可だが設計思想が極めて有用。
- 最も有用な設計思想:
  - **ログベース評価**: 実行ログからフィットネスを推定 → 殿の「評価スコアの数値化に不安」への最も直接的な解決策
  - **セマンティックGA**: LLMによる意味を保持した変異/交叉 → instructions/*.mdの進化に直接適用可能
  - **階層的評価**: 安い評価（静的解析）→高い評価（LLM比較）の段階実行 → コスト効率向上

#### 2. マルチエージェント評価系（MARBLE, ChatDev, MetaGPT）

**MARBLE / MultiAgentBench** (ACL 2025 Main)
- マルチエージェントの協調・競合を評価するベンチマーク。star/chain/tree/graphのトポロジー比較。milestone-based KPIで3軸評価（タスクスコア・協調スコア・計画効率）。
- GitHub: https://github.com/MultiagentBench/MARBLE | License: MIT (推定)
- **統合判定: 推奨（評価指標・KPI設計の参考として）** — MARBLEの3軸評価はteam-benchのTeamScore設計と補完的。milestone-based KPIの考え方は直接借用可能。ただし短期エピソード前提のため永続組織への拡張が必要。

**ChatDev** (OpenBMB/清華大学, 25,000+ stars)
- CEO/CTO/Programmer/Tester等が自動対話しソフトウェアを生成。CompanyConfig（3つのJSON設定ファイル）変更だけで体制が変わる。
- GitHub: https://github.com/OpenBMB/ChatDev | License: Apache 2.0 (code) / CC BY NC 4.0 (data)
- **統合判定: 推奨（比較対象フレームワークとして）** — 設定ファイル3つの変更で体制変異が可能。Evolution Engineの変異対象として理想的。shogun vs ChatDev式の直接比較がteam-benchの目玉になりうる。
- 通信方式の差異: ChatDev=同期ターン制・2者間、shogun=非同期・多者間。成果物品質で比較。

**MetaGPT** (ICLR 2024, 45,000+ stars)
- SOP（Standard Operating Procedures）駆動。構造化中間成果物（PRD→設計→コード）で品質担保。Publish/Subscribe通信。
- GitHub: https://github.com/FoundationAgents/MetaGPT | License: MIT | Docker Hub: metagpt/metagpt:latest
- **統合判定: 推奨（比較対象 + SOP設計の参考として）** — SOP方式のRole定義はshogunのinstructions/*.mdと対比可能。構造化中間成果物は評価指標設計に活用可能。Docker対応済みで組込み障壁が低い。
- 理想的な実験設計: shogun vs ChatDev vs MetaGPT の3体制同一タスク比較。

**3ツール共通課題**:
- LLM API前提（Claude Max直接利用不可）→ LiteLLMプロキシまたはClaude CLIラッパーで対応
- ソフトウェア開発タスク特化（ChatDev/MetaGPT）→ タスクプールはコーディング系に限定

#### 3. ベンチマーク系（WebArena, SWE-Bench, AgentBench）

**WebArena** ★殿の最大関心事項★
- 812タスク（4つの自己ホスト型Webアプリ: Shopping, Reddit, GitLab, CMS）でWebエージェントを評価。タスク成功率で判定（部分点なし）。
- GitHub: https://github.com/web-arena-x/webarena | Verified版: https://github.com/ServiceNow/webarena-verified | License: Apache-2.0
- Docker完全対応（公式イメージあり）。WebArena Verified（2026年リリース、評価関数を人手監査で改善）。
- 最新SOTA: ~62%成功率（人間78.24%）。

- **4ドメインの具体的タスク例**:
  - Shopping: 「最後に注文が発送されたのはいつか」「最も評価の高い商品をカートに追加」
  - GitLab: 「プロジェクトのREADMEにマイルストーンを追記」「新Issueを作成してラベル付与」
  - Reddit: 「特定サブレディットの最人気投稿を検索」
  - CMS: 「在庫切れ商品を無効化」

- **team-bench統合方法**:
  1. Docker Composeで4サイトを起動（sandbox環境の一部）
  2. 812タスクから50問選定（カテゴリバランス考慮）
  3. 1世代あたり10問サンプリング → 体制A/Bに同じ10問を実行
  4. 成功数でペアワイズ比較 + 所要時間・トークン使用量も副次指標

- **統合判定: 強く推奨（最優先で組み込むべき）** — 殿の高い関心、Docker対応、決定論的評価（LLM-judge不要）、豊富なタスク、Apache-2.0。
- **障壁**: ストレージ1TB（殿のMacで確保可能か要確認）、4サイト全起動時のメモリ消費（16GB環境では厳しい可能性）、ブラウザ操作ライブラリの選定。

**SWE-Bench / SWE-Bench Verified**
- GitHubの実IssueをLLMに修正させるベンチマーク。SWE-Bench: 2,294問、Verified: 500問（人手キュレーション済み）。決定論的評価（テストパス/フェイル）。
- GitHub: https://github.com/SWE-bench/SWE-bench | License: MIT | ICLR 2024採択
- SOTA: Claude 4.5系で~72% resolve rate（2026年時点）。
- **データ汚染リスク: 深刻** — 全タスクが公開GitHubリポから収集。LLM学習データに含まれている可能性高。OpenAIが2026年に評価を停止。ただし体制間の相対比較なら同じ汚染を受けるため公平。
- **統合判定: 条件付き推奨（WebArenaの次に検討）** — 決定論的テスト、Docker完全対応、MIT。コード修正タスクでチーム分業（分析→分解→修正→QC）の評価に適する。

**AgentBench** (ICLR 2024, Tsinghua/THUDM)
- 8環境（OS, DB, KnowledgeGraph, CardGame, LTP, ALFWorld, WebShop, WebBrowsing）でLLMエージェントを評価。
- GitHub: https://github.com/THUDM/AgentBench | License: Apache-2.0
- **統合判定: 非推奨（部分流用のみ）** — 8環境中team-benchに有用なのはOS+DBの2つのみ。環境が簡素化されすぎ。WebArena/SWE-Benchの方が深い。最終更新2024年で活発な開発停止。チーム評価用に設計されていない。

#### 4. チーム生成・最適化系（ARG-Designer, MAPRO, MAP-Elites）

**ARG-Designer** (AAAI 2026 Oral) ★殿向け詳細説明あり★
- タスク記述を入力すると、最適なエージェント数・役割・通信トポロジーを自動生成する。GRUベースの条件付き自己回帰グラフ生成。
- GitHub: https://github.com/Shiy-Li/ARG-Designer | License: 未明記
- 6ベンチマークでSOTA（平均92.78%）、トークン効率はG-Designerの約半分。

- **殿向け説明**: 「タスクの内容を読んで、何人のエージェントが必要か、各エージェントにどんな役割を持たせるか、誰と誰を繋げるかを、自動で一歩ずつ決めていく仕組み」。コース料理の例え: 注文内容（タスク）を見て→一品ずつ（エージェントを一人ずつ）→前の料理を見て次を決める（前に選んだ役割を踏まえて次の役割を選ぶ）→チーム完成。

- **統合判定: 条件付き推奨（Evolution Engineの体制案自動提案に最適）**
  - shogunの各役割のinstructions.mdを役割埋め込みに変換→役割プールに登録で統合可能
  - 障壁: 永続組織向けの再学習が必要（40-60例）、非同期通信対応の拡張が必要

**MAPRO** (arXiv 2510.07475, Amazon Science)
- マルチエージェントの各プロンプトをMAP推論で同時最適化。信念伝播でK^N通りの組み合わせを多項式時間で解く。Credit Assignment（どのエージェントが失敗原因か特定）が強み。
- GitHub: 公開リポジトリなし（2026年3月時点） | License: N/A
- **統合判定: 概念のみ推奨（credit assignmentの考え方を借用）** — instructions*.mdの同時最適化に理論的には最適だが、コード未公開のため自前実装が必要。Credit Assignmentの「逆トポロジーblame伝播」はteam-benchの評価パイプラインに「どのエージェントが原因か」の帰属分析として即座に借用可能。

**MAP-Elites** (pyribs, MIT License)
- 品質多様性（Quality-Diversity）アルゴリズム。特徴空間をグリッドに分割し、各セルに最良解を保持。品質と多様性を同時追求。
- GitHub: https://github.com/icaros-usc/pyribs (v0.8.3) | License: MIT | Python 3.10+
- **統合判定: 強く推奨（殿の設計に明示的に組み込み済み）** — 殿の「世代10+でMAP-Elites移行」構想に直結。pyribsのGridArchiveに過去の体制データを投入→未探索領域の可視化→空セル狙い撃ちの体制生成が可能。
- 特徴空間の例: 次元1=エージェント数(1-15)、次元2=階層深度(1-5)、次元3=中央集権度(0-1)
- 障壁: 体制定義（数千行のMarkdown）を固定長ベクトルにエンコードする方法が必要、1解の評価=約3時間のためbatch_sizeを1-2に制限

### 統合推奨（全15ツール横断）

全ツールを横断して、team-benchへの統合優先順位:

**Tier 1（即採用）**:
- **MAP-Elites (pyribs)** — 殿の設計に明示的に組み込み済み。MIT License、Python 3.10+、即使用可能。世代10+でのアーカイブ管理に不可欠。
- **CrewAI** — 被験者として最優先。YAML宣言的定義で体制差替えが最も容易。hierarchicalモードでshogunとの直接比較が可能。

**Tier 2（Phase 2で検討）**:
- **WebArena** — 殿の最大関心事項。Docker完全対応、決定論的評価、812タスク。ただしストレージ1TB・メモリ16GB環境での動作確認が必要で、環境整備に時間がかかる。
- **SWE-Bench Verified** — 決定論的テスト、Docker対応、MIT。コード修正タスクのチーム評価に最適。データ汚染は相対比較なら許容可能。Phase 1のMVPベンチマークとして引き続き有力。
- **MetaGPT** — 比較対象フレームワーク。Docker対応済み。shogun vs ChatDev vs MetaGPTの3体制比較の一角。
- **ChatDev** — 比較対象フレームワーク。CompanyConfig変異で体制実験が自動化しやすい。
- **OpenHands** — Sandbox Runner実装の設計参考として。Docker sandbox設計を直接借用。

**Tier 3（設計思想のみ借用）**:
- **ARTEMIS** — ログベース評価（評価関数不要）とセマンティックGA（LLMベース変異/交叉）の設計思想を自前実装で取り込む。階層的評価（安い→高い）もコスト削減に有効。
- **ARG-Designer** — Evolution Engineの体制案自動提案に活用。永続組織向けの再学習データが蓄積されてから本格導入。
- **MAPRO** — Credit Assignment（どのエージェントが失敗原因か）の考え方を評価パイプラインに組み込む。コード公開待ちまたは簡略版を自前実装。
- **EvoAgentX** — TextGrad/AFlowの最適化アルゴリズムを参考に。Claude Max対応可否の検証が先決。
- **MARBLE** — 評価指標（milestone-based KPI、3軸評価）の設計参考として。
- **SiriuS** — 経験ライブラリの(input, trajectory, reward)データ構造を世代記録YAMLフォーマットに反映。
- **AutoGen** — Magnetic-One等のチームパターンを変異バリエーション元として参考。maintenance mode入りのため直接統合は非推奨。
- **Aider** — 単一エージェントベースライン。リポジトリマップのコンセプトのみ参考。

**Tier 4（不採用）**:
- **AgentBench** — 8環境中有用なのはOS+DBのみ。簡素化されすぎ、開発停止、チーム評価非対応。WebArena/SWE-Benchで代替可能。

### 殿への確認事項

1. **WebArenaのストレージ1TB**: 殿のMacでディスク空き容量1TB確保可能か？ 4サイト全起動時のメモリ消費（16GB環境では厳しい可能性）も要検証。
2. **LLM API課金の許容範囲**: ChatDev/MetaGPT/AutoGen/CrewAIはいずれもLLM API前提。Claude Maxの定額内で動かすにはLiteLLMプロキシまたはClaude CLIラッパーが必要だが、この検証工数を許容するか？ それとも成果物品質のみで比較する（LLM種類を統制しない）アプローチを取るか？
3. **被験者フレームワークの優先度**: CrewAI（最も統合しやすい）を最初の被験者とすることでよいか？ それともshogun vs shogun変異体の自己比較から始めるべきか？
4. **ARG-Designerの再学習データ**: Evolution Engineで体制案を自動生成するためにARG-Designerを使う場合、初期の学習データ（40-60例のチーム構成成功/失敗データ）をどう収集するか？ team-benchの実行履歴が蓄積されるまで待つか、合成データで始めるか？
5. **MAP-Elitesの特徴空間定義**: 体制の特徴をどの軸で測るか？ 提案: エージェント数×階層深度×中央集権度×役割専門化度の4次元。殿の直感と合うか？
