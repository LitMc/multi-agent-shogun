# team-bench

## North Star
AIエージェントチームの評価ベンチマークと自動改善フレームワークを構築し、shogunを超える次世代AIエージェント体制を実現する。

## 概要
- AIエージェントの「チーム」としての良し悪しを測るベンチマーク
- 自己教師あり学習の発想を「エージェント組織」に適用
- 既存データの一部を加工し復元問題として解かせる（SWE-Bench等と同様の発想）
- チーム向けベンチマーク + 体制改善フレームワーク

## 背景
- AIエージェント体制を作るフレームワークは多数存在する
- しかしエージェント体制の良し悪しを測る方法が存在しない
- 個々のエージェントの能力ではなく「組織」としての評価が必要

## アプローチ
- 自己教師あり学習: 教師データを人が作るかわりに既存データの一部を加工し復元問題として解かせる
- チーム評価: 個人ではなくチーム全体のパフォーマンスを測定
- 自動改善ループ: ベンチマーク結果をフィードバックして体制を自動改善

## 殿の具体イメージ（2026-03-12）
- AIエージェント体制（shogunのようなもの）を複数作る
  - 出発点はmulti-agent-shogun
  - 何らかの方法で多様な体制案をつくる
    - 一部をいじったり
    - 全く違うアプローチをとらせたり
    - 縛りをもうけたり（例:「家老なしで同じパフォーマンスを実現せよ」）
- 作り出した各体制にタスクを与えて評価
- 評価のよかった体制を採用
- このサイクルを自動で回す

## 殿が挙げた課題
- ベンチマークをどうするか
  - 既存を使うか
  - 新しく作るか
- 複数のエージェント体制を独立に動かす仕組みをどうするか
  - コンテナやPython仮想環境のように閉じていて再現性のある環境が必要

## 殿の設計方針（2026-03-12追加）
- ベンチマークを作る場合、以下を別プロジェクトに分離してもよい:
  1. **ベンチマーク手法** — チーム評価のための測定・スコアリング
  2. **自動改善フレームワーク** — ベンチマークを使って体制を自動改善するサイクル

## 殿の運用制約（2026-03-12追加・更新）
- **即時開始**: 数ヶ月先ではなく今すぐ始められること
- **コスト**: Claude Max ($200/月・定額) を活用。API不可（リモート非対応）
- **環境**: ローカル1台で十分。スケールアップ不要
- **一世代 = 一晩(12時間)**: 殿が寝ている間に1世代の改善を回す。人が指示できない時間にこそ進める
- **コーパス**: 既存の評価手法やコーパスの組み合わせでもよい

## 重要発見: Docker + Claude Max が使える（2026-03-12）
- `claude setup-token` でOAuthトークンを生成 → Dockerコンテナに環境変数で渡せる
- コンテナ内でMaxサブスクリプションとして動作（API課金なし）
- 公式devcontainer設定あり
- **影響**: 当初懸念だったDockerサンドボックスが現実的に。APIコスト爆発リスクが解消
- **並列制限**: Claude Maxは同時5セッションまで → 逐次実行で一晩2〜4体制評価可能

## 殿が見ている課題（2026-03-12）

### 課題1: 評価タスクの具体化
- 手持ちの材料ではタスクの幅も量も足りない気がする
- 既存コーパスの組み合わせでもよいが、具体化が必要

### 課題2: 評価方法の具体化
- 基本はLLM-as-a-judge
- 評価者の評価ができない問題もある
- 想定スキーム:
  - 案1: 前回のコンペで最良を勝ち取ったチームに評価させる（新チームの能力に期待）
  - 案2: プレーンのClaude Codeに評価させる（評価軸の一貫性を重視）
- **殿のアイデア: ペアワイズ比較（相対評価）**
  - 絶対評価は難しい → 「どちらがよいか」を判断する相対評価に切り替え
  - AよりBがいい、という判断は単体の評価よりやりやすい
  - Gunshi報告のEloレーティングと整合（Eloは本質的にペアワイズ比較の集積）
  - 実装: 各世代ペアに同一タスクを実行させ、LLM-as-judgeが「どちらの成果物が優れているか」を判定
- **殿のアイデア: 決定論的テスト併用**
  - LLM-as-a-judgeでなくても判定できる項目は古典的テストで評価
  - コスト削減 + 不要な不確実性の除去
  - shogunの自動テスト(bats)がお手本 — メールボックス読み書き等の基本機能をシェルスクリプトで担保
  - ハイブリッド評価: 決定論的テスト(機能・正確性) + LLM-judge(品質・創造性)

### 課題3: よくない学習を防ぐ仕組み
- 丸暗記
- judgeの評価を上げるためだけに動く
- SWE-benchでもテスト削除などで問題になっているはず

### 課題4: 次の世代を作る方法
- ほしいのは根本を覆す破壊的アイデア
- 「家老を二人にしよう」ではなく「将軍、家老、足軽N人の構成にしよう」レベル
- 相当圧をかけないと現状維持に近い案ばかりになってしまいそう
- 次の世代の多様性をなるべく担保できる仕組みがほしい
- **殿のアイデア: 順序制約方式による多様性確保**
  - 複数の候補チームを順番に作る
  - 先に作るチームほど自由に作れる
  - 後から作るチームは「前のチームと被らないこと」を条件とする
  - 例:
    - 1つめ: 自由に作る。前世代の親を引き継いで素直に改善
    - 2つめ: 1つめと被らない別アプローチ
    - 3つめ: 他の手法に逆らう。常識や前提を疑う。破壊的な変更こそ命
  - 「被らない」の定義は要検討

### 課題5: 各世代・各チームの記録
- たとえ採用に至らなくとも、どんな試みをしたのかは記録しておきたい
- 過去の取り組みから知見が得られることもある → 次世代チームの作成や運用に平等に活かす
- 同じ失敗をしないための失敗データベースとしての側面もある

### 課題6: 作業の途中停止・堅牢性
- 作業が途中で止まらないか — これ自体が評価項目になりうる
- 体制をきちんと評価するためにも、与えたタスクはやりきってほしい（品質はともかく途中停止は避けたい）
- 堅牢性評価: 不意の中断からの再開・復旧もベンチマークに組み込む？
  - shogunの既存課題（Gunshiフリーズ、nudge不達等）がまさにこの領域

### 課題7: 単一評価者の危険性と対処
- Opusの問題点をOpusが見つけられるだろうか？
- 作業者と評価者で同じモデルを使う危うさ
- かといってHaikuなどスペックを落としたくはない
- GPT, Geminiなど複数の全く異なるモデルが理想だが……
  - コストが膨らむ
  - Max相当のプランに複数入る必要がある

## 1. 既存ベンチマーク比較表

| ベンチマーク | 評価対象 | チーム評価？ | 手法 | スコアリング | 主な制限 |
|---|---|---|---|---|---|
| **SWE-Bench** | 単一エージェントのGitHub issue修正 | ❌ 個人 | fail-to-pass テストパッチ生成 | Resolve Rate (%) | データ汚染リスク、協調テストなし |

| **GAIA** | 単一AIアシスタントのツール活用推論 | ❌ 個人 | 3難易度450問、マルチステップ | Accuracy (%) | 検証セット漏洩問題 |
| **AgentBench** | 単一LLMの8環境操作 | ❌ 個人 | OS/DB/Web/ゲーム等8環境 | 環境別成功率 | 環境が簡素化されている |
| **MLE-Bench** | 単一エージェントのKaggle ML課題 | ❌ 個人 | 75 Kaggle課題の自律実行 | メダル獲得率 | 計算コスト大 |

| **ChatDev / MetaGPT** | マルチエージェントソフトウェア開発チーム | ✅ チーム（框架固有） | CEO/CTO/PG/Tester等の役割分担 | 完全性・実行可能性・品質 | 標準化されていない、フレームワーク固有 |

### 差別化ポイント

**既存ベンチマークの空白地帯**（=team-benchが埋めるべき領域）:

1. **永続的階層組織の評価**: 既存は短期エピソード（数分〜数時間）。shogunのような日/週単位で稼働する永続組織は未評価
2. **非同期メールボックス通信の評価**: 既存はsync/turn-based。async inbox方式の効率は未測定
3. **品質管理チェーン（QC→redo）の評価**: ashigaru→gunshi→karo の多段QCループは既存ベンチに存在しない
4. **組織メモリ（MCP等）を持つチームの評価**: セッション跨ぎの知識保持は未測定
5. **指揮系統（chain of command）の最適化**: flat/DAGトポロジーの研究は多いが、固定階層内での最適化は未開拓

### 2.5 新規性の評価

**既に存在するもの:**
- 個別エージェント最適化（DSPy, TextGrad, ADAS）— 成熟
- ワークフロー/トポロジー最適化（MASS, AFlow, AgentConductor）— 2025-2026活発
- マルチエージェント評価インフラ（MASEval, MAESTRO, MARBLE）— 黎明期
- 統合進化フレームワーク（EvoAgentX）— 存在するが発展途上

**team-benchの新規性が成立する領域:**

## 3. 自己教師あり学習のチーム評価への適用設計

### 3.1 定式化

殿のビジョン「既存データの加工→復元問題」をチーム評価に適用する:

**基本アイデア**: shogun体制の過去の成功タスク履歴（queue/reports/*, git log等）を使い、タスクの一部を隠して復元させる。

**具体的定式化案:**

#### 方式A: タスク分解復元（Decomposition Reconstruction）
```
入力: cmd_XXXの概要（殿の指示文）
隠す: Karoによるサブタスク分解結果
課題: チームにcmd_XXXを与え、サブタスク分解→実行→QCまで実行させる
評価: 実際の分解結果との一致度 + 成果物品質
```

#### 方式B: 成果物復元（Deliverable Reconstruction）
```
入力: タスクYAML（description + acceptance_criteria）
隠す: 最終成果物（コード変更、設計文書等）
課題: チームにタスクを実行させる
評価: 成果物の品質（テスト通過率、ビルド成功、スコープ一致）
```

#### 方式C: QC判定復元（Quality Judgment Reconstruction）
```
入力: 足軽の成果物（意図的に欠陥を仕込んだもの）
隠す: 軍師のQC判定結果
課題: チームのQCプロセスで欠陥を検出できるか
評価: 欠陥検出率、false positive/negative率
```

### 3.2 shogun体制への具体適用

| 評価タスク | 測定対象 | 自己教師あり要素 |
|---|---|---|
| 過去cmdの再実行 | チーム全体の完遂能力 | 過去の成果物を正解データとして利用 |
| 意図的欠陥注入 | QCチェーンの検出力 | 既知の欠陥パターンを自動生成 |
| 指示曖昧化テスト | 解釈力・質問力 | 過去の明確な指示を曖昧化して与える |
| 並行タスク耐性 | ボトルネック測定 | タスク数を段階的に増加させる |

### 3.3 評価スコア設計

```
TeamScore = w1 * TaskCompletion + w2 * QualityScore + w3 * Efficiency + w4 * Resilience

TaskCompletion = (completed_tasks - redone_tasks) / assigned_tasks
QualityScore = qc_pass_rate * (1 - defect_escape_rate)
Efficiency = 1 / (normalized_api_cost * normalized_wall_time)
Resilience = recovery_success_rate * (1 / mean_recovery_time)
```

**「良い体制」の定義:**
- 高 TaskCompletion（タスクを確実に完了する）
- 高 QualityScore（品質管理が機能している）
- 高 Efficiency（コスト・時間効率が良い）
- 高 Resilience（障害から素早く復帰する）

重み `w1-w4` は殿の優先事項に応じて調整可能。

### 4.2 コンポーネント詳細

#### A. Benchmark Suite（ベンチマーク）

**Task Pool**:
- SWE-Bench Verified のサブセット（コード修正タスク）
- 自己生成タスク（過去のshogun作業履歴からの復元問題）
- 合成タスク（タスク分解・QC・エラーリカバリーの専用問題）

**Evaluator**:
- 成果物評価（テスト通過、ビルド成功、差分品質）
- プロセス評価（通信効率、redo率、ボトルネック検出）
- DyLAN式 Agent Importance Score（各役割の貢献度）

**Scorer**:
- TeamScore = 上記4軸の加重和
- 体制間のEloレーティング（対戦形式の相対評価）

#### B. Evolution Engine（自動改善）

**Team Variant Generator**:
- **変異**: instructions/*.md の一部をLLMで書き換え（役割定義、QC基準、通信ルール等）
- **交叉**: 2つの体制の良い部分を組み合わせ
- **制約付き生成**: 「家老なし」「軍師2体」「足軽3体のみ」等の構造制約
- ARG-Designer式の自己回帰グラフ生成も候補

**Sandbox Runner**:
- Docker コンテナ内に tmux + Claude Code 環境を構築
- 各体制を独立コンテナで並行実行
- 実行時間・トークン消費を計測
- コンテナ間の完全隔離（ファイルシステム・ネットワーク）

**Score Collector + Selector**:
- 各体制のTeamScoreを収集
- 上位N体制を次世代の親として選択（遺伝的アルゴリズム）
- MCTS（AFlow式）またはベイジアン最適化も候補

### 4.3 実行環境設計

**選択肢比較:**

| 方式 | 隔離性 | 再現性 | コスト | 実装難度 |
|---|---|---|---|---|
| **Docker コンテナ** | ◎ 完全隔離 | ◎ Dockerfile固定 | △ コンテナ×体制数 | ○ 中 |
| **tmux セッション分離** | △ ファイルシステム共有 | △ OS状態に依存 | ◎ 追加コストなし | ◎ 低 |
| **git worktree** | ○ ファイル隔離 | ○ git管理 | ◎ 追加コストなし | ○ 中 |
| **VM / クラウドインスタンス** | ◎ 完全隔離 | ◎ イメージ固定 | ✕ 高コスト | △ 高 |

**推奨: Docker コンテナ**
- 理由: 完全隔離 + 再現性 + 殿のOCI環境（4 OCPU / 24G RAM）で並行実行可能
- 各コンテナに: tmux + Claude Code + テスト対象リポジトリ + 体制定義ファイル
- API keyは環境変数で注入（コンテナ内に埋め込まない）

### 4.4 体制変異の具体手法

```
Original体制 (multi-agent-shogun)
  │
  ├── 変異A: instructions/karo.md のQCルールを緩和
  ├── 変異B: 軍師を削除、足軽がセルフQC
  ├── 変異C: 足軽を3体に削減
  ├── 変異D: 通信をsync方式に変更
  ├── 変異E: Bloom routing閾値を変更
  ├── 変異F: 全エージェントをOpusに統一
  ├── 変異G: フラット構造（shogun直下に全員）
  └── 変異H: LLM生成による完全新規体制
```

**変異の自動化**: ADAS式メタエージェントが体制定義（instructions/*.md + config/settings.yaml）を読み、変異版を自動生成。TextGrad式テキスト勾配で方向性を持った改善。

## 5. プロジェクト分離の推奨方針

### 選択肢分析

| 構成 | メリット | デメリット |
|---|---|---|
| **A: 統合（1リポジトリ）** | コード共有容易、依存管理シンプル | スコープ肥大化、貢献者混乱 |
| **B: 2分離** (bench + evolver) | 関心の分離明確、独立リリース可 | リポジトリ間API契約の維持コスト |
| **C: 3分離** (bench + evolver + sandbox) | 最大の柔軟性 | 過剰分離、初期段階では複雑すぎ |

### ★推奨: 方式B — 2リポジトリ分離

```
team-bench/          ← ベンチマーク手法（タスクプール + 評価器 + スコアラー）
team-evolver/        ← 自動改善フレームワーク（変異生成 + サンドボックス + 選択）
```

**根拠:**
1. **ベンチマーク単体で価値がある**: 他のマルチエージェントフレームワーク（AutoGen, CrewAI等）のユーザーもteam-benchで自分の体制を評価できる
2. **進化エンジンはベンチマークに依存するが逆は不要**: team-benchは独立して使える
3. **OSSとしての採用率**: ベンチマーク単体の方が採用されやすい（MASEval, MARBLE等の先例）
4. **初期段階では統合でも可**: まずmonorepo内で開発し、安定したらリポジトリを分離する段階的アプローチ

**初期実装ロードマップ:**
```
Phase 0: monorepo (team-bench/) 内で両方開発     ← 今ここ
Phase 1: Benchmark Suite の MVP（SWE-Benchサブセット + プロセス評価）
Phase 2: Sandbox Runner MVP（Docker 1体制実行）
Phase 3: Team Variant Generator MVP（手動変異 → 自動変異）
Phase 4: リポジトリ分離（API契約が安定したタイミング）
```

## 6. shogun体制の強み・弱み分析

### 強み
1. **品質管理チェーン**: ashigaru→gunshi QC→karo判定の多段チェック。redo率低減に寄与
2. **並列実行**: 足軽7体の並行作業で高スループット
3. **イベント駆動**: ポーリング不要、コスト効率が高い
4. **コンテキスト最適化**: YAML slimming、Bloom routing、自己/clear
5. **耐障害性**: エスカレーション3段階、inbox永続化、atomic書き込み
6. **追跡可能性**: 全アクションにtimestamp、parent_cmd、bloom_level

### 弱み
1. **Karoボトルネック**: 全タスクがKaroを経由。Karo不在時に全軍停止
2. **Gunshi単一**: QCが逐次処理。7足軽同時完了時にQC待ち行列発生
3. **足軽間知識非共有**: 各足軽は独立。他の足軽の知見を活用できない
4. **固定スケール**: 足軽7体が上限。動的スケーリング不可
5. **指示書の認知負荷**: 全体で2,400行超。新エージェントの立ち上げコスト大
6. **エラーリカバリーの手動性**: 足軽クラッシュ時の再割り当てはKaro手動介入

### team-benchで測定すべき指標候補

| 指標 | 計算式 | 測定対象 |
|---|---|---|
| タスク完遂率 | completed / assigned | チーム全体の信頼性 |
| QC通過率 | gunshi_pass / qc_total | 足軽の品質 |
| Redo率 | redo_tasks / total_tasks | タスク分解・指示の精度 |
| メッセージ効率 | messages / completed_task | 通信オーバーヘッド |
| 未読遅延 P95 | inbox_write → agent_read 時間 | 通知システムの信頼性 |
| APIコスト/タスク | token_spend / completed_task | コスト効率 |
| 壁時計完了時間 P50/P95 | cmd投入→全subtask完了 | エンドツーエンド性能 |
| Karo利用率 | karo_busy_time / total_time | ボトルネック検出 |
| 障害復帰時間 MTTR | failure→resume 時間 | 耐障害性 |

## 課題1対策: タスクプール設計

### 1.1 タスクソースの3層構造

| 層 | ソース | 目的 | タスク数目安 |
|---|---|---|---|
| **L1: 自己教師あり** | shogun作業履歴（queue/reports/*, git log） | 自チーム固有能力の測定 | 10-15問 |
| **L2: 外部コーパス** | SWE-Bench Verified サブセット + HumanEval変種 | 汎化能力の測定 | 15-20問 |
| **L3: 合成タスク** | チーム協調専用の自動生成問題 | 組織能力（通信・QC・復帰）の測定 | 10-15問 |

**一晩（12時間）2〜4体制評価に必要な規模:**
- Claude Max 5セッション同時制限 → 1体制あたり最大5セッション使用
- 逐次実行: 1体制あたり3時間（12h ÷ 4体制）
- 3時間で処理可能なタスク数: 8-12問（タスク難易度による）
- **推奨: 1世代あたり10問 × 4体制 = 40タスク実行**（タスクプール全体から10問をサンプリング）

### 1.2 L1: 自己教師ありタスクの具体例

| # | タスク種別 | 入力 | 隠す部分 | 難易度 |
|---|---|---|---|---|
| 1 | cmd分解復元 | cmd概要文 | Karoのsubtask分解 | 中 |
| 2 | 成果物復元 | task YAML | 最終コード差分 | 高 |
| 3 | QC欠陥検出 | 欠陥注入済み成果物 | 欠陥の位置と種類 | 中 |
| 4 | 曖昧指示解釈 | 曖昧化したcmd | 元の明確な指示 | 高 |
| 5 | 通信効率テスト | 依存関係のある3subtask | 最小メッセージ数 | 低 |

**生成パイプライン:**
```
git log --since="2026-01-01" queue/reports/
  → 成功タスク抽出
  → LLMで「隠す部分」を自動選定
  → 正解データと評価基準を自動生成
  → タスクプールに追加
```

### 1.3 L2: 外部コーパスの選定基準

- **SWE-Bench Verified**: 500問中、Python単体ファイル修正で30分以内に解ける問題を50問選定 → 1世代で10問サンプリング
- **選定条件**: テスト付き・再現可能・Dockerコンテナ内で自己完結
- 短時間（15-30分/問）で決定論的に正誤判定可能なものを優先

### 1.4 L3: 合成タスクのカテゴリ

| カテゴリ | 測定対象 | 例 |
|---|---|---|
| 分解テスト | タスク分解精度 | 「100行のバグ修正を3サブタスクに分解せよ」 |
| QCテスト | 品質管理能力 | 「3つの成果物をレビューし欠陥を報告せよ」 |
| 復帰テスト | 耐障害性 | 途中でエージェントをkill → 復旧測定 |
| 並行テスト | ボトルネック検出 | 5タスク同時投入 → 完了順序と所要時間 |

### 1.5 難易度分布

```
Easy  (15分以内): 30% — 基本機能テスト、単体ファイル修正
Medium(30分以内): 50% — 複数ファイル修正、タスク分解必要
Hard  (60分以内): 20% — アーキテクチャ変更、複数エージェント協調必須
```

この分布により、弱いチームでも部分点を取れ、強いチームが差別化できる。

## 課題2対策: ハイブリッド評価設計

### 2.1 評価の2層構造

```
┌─────────────────────────────────────────┐
│          ハイブリッド評価パイプライン          │
│                                          │
│  ┌─────────────────┐  ┌──────────────┐  │
│  │ Tier 1:          │  │ Tier 2:       │  │
│  │ 決定論的テスト    │  │ LLM-judge     │  │
│  │ (bats/pytest等)  │  │ (ペアワイズ)    │  │
│  │                  │  │               │  │
│  │ ・テスト通過率    │  │ ・コード品質   │  │
│  │ ・ビルド成功     │  │ ・設計妥当性   │  │
│  │ ・lint/型チェック │  │ ・通信効率の   │  │
│  │ ・inbox形式検証  │  │   質的評価     │  │
│  │ ・復帰成功率     │  │ ・指示解釈の   │  │
│  │                  │  │   適切さ       │  │
│  │ → 0/1 確定スコア │  │ → 比較判定     │  │
│  └────────┬────────┘  └──────┬───────┘  │
│           │                    │          │
│           └──────┬─────────────┘          │
│                  ▼                        │
│         統合スコア算出                      │
│  Score = α * Tier1 + (1-α) * Tier2_Elo   │
│  推奨 α = 0.6 (決定論的テスト重視)          │
└─────────────────────────────────────────┘
```

### 2.2 Tier 1: 決定論的テスト項目

| テスト項目 | 測定方法 | 判定基準 | ツール |
|---|---|---|---|
| **タスク完了** | subtask全完了チェック | 全subtask status=done | bats |
| **ビルド成功** | `npm run build` / `make` | exit code 0 | CI script |
| **テスト通過** | `pytest` / `bats` | pass率 100% (SKIP=FAIL) | pytest/bats |
| **lint通過** | `shellcheck` / `eslint` | warning 0 | lint tools |
| **inbox形式** | YAML schema validation | 必須フィールド存在 | bats + yq |
| **report形式** | YAML schema validation | 必須フィールド存在 | bats + yq |
| **復帰成功** | kill → 自動復帰チェック | 120秒以内にresume | bats + tmux |
| **ファイル存在** | 成果物ファイル存在確認 | 全deliverables存在 | bats |

**Tier 1のカバー率推定: 全評価の60-70%** — 機能正確性・形式遵守・基本動作はテストで完全判定可能。

### 2.3 Tier 2: LLM-judgeペアワイズ比較

**基本方式**: 体制Aと体制Bの成果物を同一LLM-judgeに並べて提示し、「どちらが優れているか」を判定。

**ペアワイズ比較の実装:**
```
入力: {task_description, output_A, output_B}
判定: A >> B | A > B | A ≈ B | B > A | B >> A
  (>>: 明確に優位, >: やや優位, ≈: 同等)
スコア変換: A>>B → A+3,B+0 | A>B → A+2,B+1 | A≈B → A+1.5,B+1.5
```

**Eloレーティングへの統合:**
- 各ペアワイズ結果をEloの勝敗として処理
- K-factor = 32（初期）→ 16（安定期）
- 10問 × 4体制 = 6ペア比較/問 × 10問 = 60対戦 → 統計的に有意なElo差を検出可能

**ペアワイズで評価する項目:**

| 項目 | 比較観点 | 判定ポイント |
|---|---|---|
| コード品質 | 可読性・保守性・命名 | 同一タスクの2成果物を比較 |
| 設計妥当性 | アーキテクチャ判断の質 | 設計レポートを比較 |
| 通信効率 | メッセージの明確さ・無駄のなさ | inboxログ全体を比較 |
| エラーハンドリング | 異常時の判断の質 | 障害注入時のログを比較 |

### 2.4 評価者の選定: ★推奨案

| 方式 | メリット | デメリット | 推奨度 |
|---|---|---|---|
| **案1: 前世代最良チーム** | 実戦経験あり、高い判断力 | 自身に有利なバイアス可能性 | △ |
| **案2: プレーンClaude Code** | 一貫性・中立性が高い | チーム動態の文脈理解が浅い | ○ |
| **★案3: ペアワイズ専用judge** | Tier 1テストが客観軸を担保、judgeは質的比較に集中 | judge設計コスト | ◎ |

**★推奨: 案3 — ペアワイズ専用judge + Tier 1テスト併用**

**根拠:**
1. Tier 1テスト（決定論的）が「正しいか/動くか」を完全に判定 → judgeに客観性の負担をかけない
2. judgeは「どちらが質的に優れているか」のみに集中 → 判断負荷が低く精度が安定
3. ペアワイズ比較は単体評価より再現性が高い（先行研究でも確認済み）
4. プレーンClaude Code（テンプレートプロンプト付き）をjudgeとして使用 → コスト最小

**評価者の評価（meta-evaluation）問題への対策:**
- **Position bias対策**: A/Bの提示順をランダム化（同一ペアを2回、順序入替で判定）
- **一貫性検証**: 同一ペアの2回判定が矛盾したら無効化して再判定
- **Tier 1との整合性**: Tier 1テストで明らかにA>Bなのにjudgeが逆判定 → 異常フラグ

## 課題3対策: 過学習・ハック防止機構

### 3.1 脅威モデル

| ハック種別 | 具体例 | SWE-Benchでの事例 |
|---|---|---|
| **丸暗記** | 正解パッチをメモリから復元 | データ汚染（学習データにSWE-Bench含有） |
| **テスト削除** | fail-to-passテストを削除してpass | SWE-Bench GitHub issue多数報告 |
| **judge最適化** | 評価観点を逆手にとったスコア稼ぎ | LLM-as-judge研究の既知問題 |
| **メトリクスゲーム** | メッセージ数を人為的に減らし効率偽装 | 本システム固有リスク |
| **自己改竄** | 評価スクリプト自体を改変 | サンドボックス脆弱性として一般的 |

### 3.2 防止機構

#### A. タスクプールの動的ローテーション

```
世代1: タスクセットA（10問からランダム5問）
世代2: タスクセットB（別の10問からランダム5問）
世代3: タスクセットA' (Aの変形版: 変数名変更、要件微調整)
...
```
- 同一タスクの連続使用を禁止 → 丸暗記の価値を激減
- タスク変形パイプライン: LLMで変数名・関数名・要件の非本質部分を自動変更

#### B. テスト保護（Read-Only Test Suite）

```
# Dockerコンテナの構成
/workspace/          ← チームが自由に操作可能
/tests/              ← Read-Only マウント（コンテナ内から書き込み不可）
/evaluator/          ← Read-Only（評価スクリプト）
```
- テストファイルをread-onlyでマウント → 削除・改変不可能
- テスト実行は評価パイプライン側が行う（チーム側からは呼び出し不可）

#### C. judge操作防止

- **ブラインド評価**: judgeはチームIDを知らない（匿名化された成果物を比較）
- **多観点ペアワイズ**: 1つの観点だけ最適化しても総合スコアが上がらない構造
- **Tier 1テストの重み**: α=0.6 により、決定論的テストが過半を占める → judge操作の利得が限定的

#### D. メトリクスゲーム防止

- **通信効率**: メッセージ数だけでなく「タスク完了あたりのメッセージ内容量」で評価 → 空メッセージ水増しに無意味
- **完了時間**: 壁時計時間で測定。途中停止は即座に減点 → 省略による高速化が不可能

#### E. サンドボックス保護


## 課題4対策: 構造的多様性メカニズム ★最重要★

### 4.1 殿の順序制約方式の詳細設計

**基本原理**: N個の候補チームを逐次生成し、後発ほど強い差別化制約を課す。

**具体的実装:**

```yaml
# team-evolver の generation config
generation:
  teams_per_generation: 4  # 一晩4体制（12h ÷ 3h/体制）
  slots:
    - slot: 1
      label: "素直な改善"
      constraint: |
        前世代の最良チームをベースに自由に改善せよ。
        変更範囲に制限なし。
      diversity_pressure: low

    - slot: 2
      label: "別アプローチ"
      constraint: |
        slot1のチーム構成を参照し、以下の条件を満たせ:
        - slot1と同一の役割構成を使用禁止
        - slot1が採用した通信方式と異なる方式を使用
        - 最低1つの役割を追加または削除すること
      diversity_pressure: medium
      diff_target: slot1

    - slot: 3
      label: "破壊的変更"
      constraint: |
        slot1, slot2の共通前提を列挙し、そのすべてに逆らえ:
        - 階層構造を使うなら→フラットにせよ
        - 専門化を使うなら→汎用化せよ
        - 中央管理なら→分散化せよ
        - 非同期なら→同期にせよ
        最低3つの「常識」を破壊すること。
      diversity_pressure: high
      diff_target: [slot1, slot2]

    - slot: 4
      label: "ワイルドカード"
      constraint: |
        以下のいずれかをランダムに選択:
        a) 全く異なるドメイン（軍事、スポーツ、工場）のメタファーで構成
        b) エージェント数を極端に変更（1体 or 15体）
        c) 前世代の最弱チームの構成を逆転の発想で改良
        d) 既存研究(MASS, AgentNet等)の構成を忠実に再現
      diversity_pressure: extreme
```

### 4.2 「被らない」の定義 — 構造差分メトリクス

チーム構成を特徴ベクトルに変換し、距離を測定:

```python
# チーム構成の特徴ベクトル（8次元）
def team_vector(config):
    return {
        "n_agents":        len(config.agents),            # エージェント数
        "n_roles":         len(set(a.role for a in config.agents)),  # 役割種類数
        "hierarchy_depth": config.max_depth,               # 階層の深さ
        "centralization":  config.bottleneck_ratio,        # 中央集権度 (0-1)
        "comm_sync":       1 if config.comm == "sync" else 0,  # 通信方式
        "has_qc":          1 if config.qc_chain else 0,    # QC機構の有無
        "has_memory":      1 if config.shared_memory else 0,  # 共有メモリの有無
        "specialization":  config.role_entropy,            # 役割の専門化度 (entropy)
    }

# 差分距離
def structural_distance(team_a, team_b):
    va, vb = team_vector(team_a), team_vector(team_b)
    return cosine_distance(va, vb)

# 制約: slot N のチームは全先行slotとの最小距離を保証
MIN_DISTANCES = {2: 0.3, 3: 0.6, 4: 0.5}
```

**差分チェックの実装:**
- slot生成後、先行slotとの構造距離を自動計算
- 閾値未満 → 生成LLMに「もっと大きく変えろ」とフィードバックして再生成（最大3回）
- 3回再生成しても閾値未達 → 制約を1段緩和して再試行

### 4.3 他の多様性担保手法との比較

| 手法 | 概要 | メリット | デメリット | team-benchへの適合 |
|---|---|---|---|---|
| **★順序制約方式（殿案）** | 後発ほど差別化条件を強化 | 実装シンプル、確実に多様性が出る、人間が理解可能 | N増加時に制約設計が困難 | ◎ 4チーム程度に最適 |
| **MAP-Elites** | 特徴空間をグリッド分割、各セルの最良を保持 | 特徴空間の均一カバー | 特徴次元の設計が必要、収束が遅い | ○ 長期運用（10世代+）に向く |
| **Quality-Diversity (QD)** | 品質と多様性を同時最適化 | 理論的に最強の多様性保証 | 計算コスト大、実装複雑 | △ コスト制約に合わない |
| **制約付きランダム** | ランダム生成 + 最小距離制約 | 実装最シンプル | 品質が低い、ゴミが多い | △ 初期探索のみ |
| **遺伝的交叉** | 2親から子を生成 | 良い特性の組み合わせが得られる | 多様性が世代とともに収縮 | ○ 順序制約と併用推奨 |
| **島モデル** | 独立に進化する複数集団 | 並列性が高い、局所解脱出 | コスト線形増加 | △ 1台ローカルでは困難 |

### 4.4 推奨: 順序制約 + 遺伝的交叉のハイブリッド

**Phase 1（世代1-3）**: 順序制約方式で広く探索
- 4スロットの制約付き生成で多様な体制を作る
- 各世代の最良スコアを記録

**Phase 2（世代4+）**: 順序制約 + 交叉
- slot1: 前世代最良2体制の交叉（良い特性の組み合わせ）
- slot2: slot1と「被らない」制約
- slot3: 破壊的変更（前世代の共通前提を否定）
- slot4: ワイルドカード

**Phase 3（世代10+）**: MAP-Elites移行
- 十分な体制データが蓄積したら特徴空間を構築
- 未探索セルに向けた狙い撃ち生成

1. **エリート保護上限**: 前世代からの「そのまま持ち越し」は最大1体制 → 3/4は新規

## 課題6対策: タスク途中停止の防止と堅牢性評価

### 6.1 堅牢性を評価項目として組み込む

**現状の課題**: shogunでもGunshiフリーズ、nudge不達が発生。途中停止は「バグ」ではなく「体制の弱点」として測定すべき。

**堅牢性スコアの定義:**

```
Resilience = w_a * CompletionRate + w_b * RecoveryRate + w_c * RecoverySpeed

CompletionRate = tasks_completed_without_interruption / total_tasks
RecoveryRate = successful_recoveries / total_interruptions
RecoverySpeed = 1 / mean_recovery_time_seconds (正規化)

推奨重み: w_a=0.4, w_b=0.35, w_c=0.25
```

### 6.2 故意障害注入（Chaos Engineering）

**注入する障害の種類:**

| 障害種別 | 注入方法 | 測定対象 |
|---|---|---|
| **エージェント停止** | `tmux send-keys -t {pane} "" C-c` → 10秒後に観測 | 他エージェントが検知→復旧できるか |
| **inbox遅延** | inbox_watcher の sleep を 30秒に変更 | 通信遅延下での作業継続 |
| **メモリ圧迫** | コンテナメモリを4GBに制限 | コンテキスト枯渇下の動作 |
| **ファイル消失** | タスク実行中にreport YAMLを削除 | データ損失からの復旧 |
| **偽メッセージ** | 不正なinboxメッセージを注入 | メッセージ検証・拒否能力 |

**注入タイミングのプロトコル:**
```
1. タスク開始後、完了までの中間地点（50%時点）で1回注入
2. 注入後120秒以内に復旧 → pass
3. 120秒超で復旧 → partial (復旧時間を記録)
4. 復旧できず → fail
5. 各タスクの10%で障害注入（全タスクではない — コスト制約）
```

### 6.3 途中停止防止の体制側メカニズム

team-benchが評価する「良い堅牢性設計」のパターン:

| パターン | 実装例 | 効果 |
|---|---|---|
| **ウォッチドッグ** | 各エージェントに60秒ハートビート | 停止を素早く検知 |
| **自動再起動** | supervisor が停止検知→/clear送信 | 人手不要の復旧 |
| **チェックポイント** | タスク進捗をYAMLに逐次保存 | 途中から再開可能 |
| **冗長エージェント** | 同一役割を2体で待機 | 1体停止でも継続 |
| **グレースフルデグレード** | QC停止時→セルフQCにフォールバック | 完全停止を回避 |

## 課題7対策: 評価者多様性確保（コスト制約: Claude Max定額のみ）

### 7.1 問題の本質

- 作業者（Opus）と評価者（Opus）が同一モデル → 共通の盲点を持つ
- 理想はGPT-4o/Gemini等のマルチモデル評価だが、複数サブスク加入のコスト問題
- **制約**: Claude Max ($200/月) のみ

### 7.2 コスト制約内での多様性確保策

#### A. Tier 1テストの最大活用（★最重要）

**ハイブリッド評価（課題2対策）が最大の緩和策:**
- Tier 1（決定論的テスト）は「モデルの盲点」と無関係 — bats/pytestに盲点はない
- α=0.6 で全評価の60%がモデル非依存 → 単一モデルリスクが大幅に低減
- **実質的にLLM-judge依存は40%のみ**

#### B. プロンプトペルソナの多様化

同一Opusでも、プロンプトで評価視点を変える:

```yaml
judge_personas:
  - id: pragmatist
    prompt: |
      あなたは10年の実務経験を持つプラグマティックなエンジニア。
      「動くか」「保守できるか」「コストは妥当か」を最重視する。
      美しさや理論的優雅さより実用性を優先せよ。

  - id: architect
    prompt: |
      あなたは分散システムのアーキテクト。
      設計の整合性、拡張性、障害耐性を最重視する。
      短期的な動作よりも長期的な構造の健全性を優先せよ。

  - id: adversary
    prompt: |
      あなたはセキュリティ研究者兼品質保証のスペシャリスト。
      エッジケース、障害シナリオ、ハック可能性を最重視する。
      「壊せるか？」「ズルできるか？」の視点で評価せよ。
```

- 各ペアワイズ比較を3ペルソナで独立判定 → 多数決
- ペルソナ間の不一致率が高い場合 → 「判断困難」フラグ → 追加テストで補強

#### C. Temperature多様化

```yaml
judge_runs:
  - temperature: 0.0  # 決定論的判断
  - temperature: 0.3  # やや探索的
  - temperature: 0.7  # 探索的
```
- 同一プロンプト・同一入力でtemperatureを変えて3回判定 → 多数決
- 3回とも一致 → 高信頼度判定
- 2:1分裂 → 中信頼度
- 全不一致 → 低信頼度 → Tier 1スコアのみで決定

#### D. 役割ローテーション（世代間）

- 奇数世代: ペルソナA(pragmatist)が主judge
- 偶数世代: ペルソナC(adversary)が主judge
- 特定ペルソナへの過適応を防止

#### E. 人間スポットチェック（低コスト）

- 各世代のjudge判定結果から、**信頼度が最低の5件**を殿に提示
- 殿の判断と一致 → judge信頼度を確認
- 不一致 → judgeプロンプトの調整材料に
- 1世代あたり5件の確認 → 殿の負担は最小限（朝起きて5分で確認可能）

### 7.3 多様性確保策の効果見積もり

| 施策 | 盲点緩和効果 | 追加コスト | 実装難度 |
|---|---|---|---|
| Tier 1テスト（60%） | ◎ モデル盲点と完全無関係 | なし | 低 |
| ペルソナ多様化（3種） | ○ 視点の偏り軽減 | 3倍のjudge実行 | 低 |
| Temperature多様化 | △ 微小な判断揺れ | 3倍のjudge実行 | 最低 |
| 役割ローテーション | ○ 世代間の過適応防止 | なし | 低 |
| 人間スポットチェック | ◎ 究極のグラウンドトゥルース | 殿の5分/世代 | 低 |

**推奨組み合わせ**: Tier 1テスト + ペルソナ多様化 + 人間スポットチェック
- ペルソナ3種 × 各判定1回 = judge実行3回/ペア → コスト許容範囲内
- Temperature多様化は追加オプション（コストが許せば併用）
